# 08 需要量予測・配送先按分 設計書

## 1. 目的・背景

現行の VRP システムは `delivery_transaction.csv` を手動でアップロードすることを前提としており、
「明日何件の配送が発生するか」を事前に把握する手段がない。

本設計では需要量予測モデルを追加し、予測件数を神奈川県の住所プールへ按分することで
`delivery_transaction.csv` を自動生成する仕組みを構築する。
これにより、VRP パイプラインへの入力生成を含めた**一気通貫の配送計画パイプライン**が完成する。

---

## 2. システム全体像

```text
【住所プール拡充】（初回のみ・モデル完成後に実施）
fetch_kanagawa_poi.py（TARGET_COUNT 拡大 + タグ追加）
    ↓
geocode.py
    ↓
delivery_destination_master.json（住所プール）

【本番フロー（日次）】

① 需要量予測
   Olist 時系列データで学習したモデル
       → 「明日の配送件数: N 件」を出力
           ↓
② 配送先按分
   住所プールから N 件をサンプリング
   package_count・delivery_time_slot_code を分布から付与
       → delivery_transaction.csv を自動生成
           ↓
③ VRP 最適化（既存パイプライン・無改修）
   geocode → snap → OD 行列 → ソルバー → 評価 → 可視化
```

## 3. ステップ①：需要量予測

### 3.1 予測対象

| 項目 | 内容 |
|---|---|
| 予測変数 | 翌日の配送件数（件/日） |
| 予測単位 | デポ単位（= デポが担当するエリアの需要量。デポごとに独立したモデルを持つ） |
| 予測タイミング | VRP 実行前（当日の配車計画を立てる前に実行する） |

### 3.2 使用データ

**Kaggle: Brazilian E-Commerce Public Dataset by Olist**
（`https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce`）

| ファイル | 使用カラム | 用途 |
|---|---|---|
| `olist_orders_dataset.csv` | `order_purchase_timestamp`, `order_status` | 日次注文件数の集計 |
| `olist_order_items_dataset.csv` | `order_id`, `product_id` | 1注文あたりの商品数（package_count 分布の参考） |

**前処理方針（最低限の前処理として以下を実施）**

1. `order_status == "delivered"` に絞り、`order_purchase_timestamp` を日付単位に集計
2. 日次件数の時系列データ（`date`, `order_count`）を生成
3. 欠損日（0件日）は 0 で補完
4. 学習期間：2016-09〜2018-08（約2年）、評価期間：直近3ヶ月

> EDA の結果によって追加の前処理（外れ値処理・変換等）が発生する場合がある。

**前提と限界**

Olist はブラジルの EC データであり、日本の物流パターン（曜日傾向・繁忙期等）と完全には一致しない。
本 PoC では「時系列予測の実装・評価プロセスを示すこと」を主目的とし、
データの地域的差異は設計上の制約として明記する。

### 3.3 特徴量

設計段階では探索する**特徴量カテゴリ**を定義する。具体的な特徴量名と採否は EDA 実施後に確定する。

| カテゴリ | 探索内容 |
|---|---|
| カレンダー情報 | 曜日・月・週番号・月初月末・連休フラグ等 |
| ラグ特徴量 | 過去 N 日の件数（N は自己相関分析で決定） |
| 移動統計量 | 移動平均・移動標準偏差（ウィンドウサイズは EDA で決定） |
| トレンド・季節性 | 長期トレンド・週次・月次季節性の有無 |

> 各モデルで有効な特徴量カテゴリが異なるため、モデルごとに適用する特徴量セットを変える場合がある。
> 最終的な特徴量構成は実装ドキュメントに記載する。

### 3.4 モデル選定

複数の候補モデルを実装・比較し、評価指標が最良のモデルを採用する。
採用判断は評価指標に加え、解釈性・実行時間も考慮する。

| モデル | 特徴 | EDA での確認事項 |
|---|---|---|
| LightGBM | 表形式特徴量に強く、特徴量重要度で解釈性を示せる | 特徴量間の相関・外れ値の影響 |
| Prophet | トレンド・季節性を自動分解。特徴量エンジニアリング不要 | トレンドの有無・季節性の周期 |
| SARIMA | 古典的手法。定常性・自己相関を前提とする | 単位根検定（ADF）・ACF/PACF |

> モデルごとに前提とするデータ特性が異なるため、EDA はモデル選定と並行して実施する。
> 比較結果は評価レポートとして出力し、採用根拠を残す。

### 3.5 評価指標

| 指標 | 説明 |
|---|---|
| MAE（平均絶対誤差） | 件数のズレを直感的に把握できる主指標 |
| RMSE（二乗平均平方根誤差） | 外れ値への感度を確認 |
| MAPE（平均絶対パーセント誤差） | スケール非依存での比較用 |

### 3.6 出力

需要量予測の出力は「予測データ」と「モデル評価結果」の2種類を保存する。

**予測データ**（日次実行時の出力）

| カラム | 内容 |
|---|---|
| `target_date` | 予測対象日 |
| `predicted_count` | 予測配送件数 |
| `model` | 使用モデル名 |
| `predicted_at` | 予測実行日時 |

**モデル評価結果**（学習・評価時の出力）

| カラム | 内容 |
|---|---|
| `model` | モデル名 |
| `MAE` | 平均絶対誤差 |
| `RMSE` | 二乗平均平方根誤差 |
| `MAPE` | 平均絶対パーセント誤差 |
| `train_period` | 学習期間 |
| `eval_period` | 評価期間 |
| `trained_at` | 学習実行日時 |

## 4. ステップ②：配送先按分

### 4.1 住所プールの拡充

**なぜ拡充が必要か**

現状の `delivery_destination_master.json` は約 100 件（`fetch_kanagawa_poi.py` の `TARGET_COUNT=100` で生成）。
予測件数がプール件数を超えると同一住所への重複サンプリングが発生し、現実的な配送計画にならない。
需要量予測の最大想定件数（例：300 件）を上回るプールサイズを確保するために拡充する。

**fetch_kanagawa_poi.py の変更点**

| パラメータ | 現行 | 変更後（目安） | 変更理由 |
|---|---|---|---|
| `TARGET_COUNT` | 100 | 500 | 最大予測件数 300 件に対して余裕を持たせるため |
| `TAGS` | 郵便局・薬局・銀行・コンビニ・電器店 | 上記 + スーパー・飲食店・病院等を追加 | 同一カテゴリのみでは地理的偏りが生じるため、多様な施設タイプで分散させる |

拡充後に `geocode.py` を実行し、`snap_status == "success"` の件数が
最大予測件数を上回ることを確認する。

### 4.2 サンプリング方式

```text
住所プール（snap_status == "success" のエントリ）
    ↓ 重複なし無作為抽出（predicted_count 件）
サンプリング済みアドレス一覧
    ↓ package_count・delivery_time_slot_code を付与
delivery_transaction.csv
```

**package_count の付与**

既存データの分布（一様分布 1〜5 個）をそのまま踏襲する。

**delivery_time_slot_code の付与**

均等割は行わない。付与方式は別途検討・記載する。

### 4.3 生成スクリプト

```text
src/vrp_optimization/demand_forecast/
├── train.py                 # モデル学習・保存
├── predict.py               # 翌日需要量の予測
└── generate_transaction.py  # 按分・CSV 生成
```

`generate_transaction.py` の出力は既存の `delivery_transaction.csv` と**同一スキーマ**とし、
VRP パイプライン側の改修なしに接続する。

将来的にはパイプライン起動時に自動で渡す構成に拡張する。


## 6. 制約・前提・今後の課題

| 項目 | 内容 |
|---|---|
| データの地域差 | Olist はブラジル EC データ。日本の物流季節性・曜日傾向と異なる可能性がある |
| 住所プール上限 | プール件数が予測件数を下回る場合は重複サンプリングが発生する。拡充で対応 |
| time_slot_code の付与方式 | 均等割は行わない。付与方式は別途検討 |
| デポ別モデル | 現行設計は単一デポ。複数デポへの対応は 09 番以降で検討 |
| 予測精度の商用利用 | PoC 目的のため概算精度で許容。実運用時は日本の実データへの置き換えが必要 |
