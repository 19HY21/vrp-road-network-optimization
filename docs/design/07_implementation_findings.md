# 実装知見と設計修正方針

`05_algorithm_design.md` および `06_experiment_design.md` の補足として位置づける。

## 1. ソルバーの CP-SAT → OR-Tools Routing Library への変更

### 1.1 変更理由

初期検証（19 配送先）では OR-Tools CP-SAT が 2.25 秒で最適解を返したため採用した（`05_algorithm_design.md` 第 2.4 節参照）。しかし 100 配送先への拡張時にタイムアウトが発生した。

CP-SAT は VRP を汎用制約プログラミングとして定式化するため、車両台数 k と地点数 n に対して **O(k × n²) のバイナリ変数**を生成する。100 配送先・k=12 では約 12 万変数となり、制限時間（120 秒）内に最初の実行可能解すら見つからなかった。

OR-Tools Routing Library は VRP 専用に設計されており、この変数爆発が起きない。ヒューリスティクスベースのため最適性証明はできないが、大規模問題でも現実的な時間で解が得られる。

### 1.2 ライブラリ特性比較

| 特性 | CP-SAT（旧） | Routing Library（現行） |
| --- | --- | --- |
| 最適性保証 | OPTIMAL ステータスで証明可能 | 不可（ヒューリスティクス） |
| 変数数 | O(k × n²) | 内部処理（変数爆発なし） |
| 小規模（〜20 件） | 高速・最適性証明付き | やや遅い |
| 大規模（100 件以上） | タイムアウト | 実用的な時間で解が得られる |
| 毎回同一の解 | 保証される | 保証されない（探索の確率的性質） |

### 1.3 CP-SAT 設計からの主な仕様変更点

**台数スイープから最大台数指定への変更**

CP-SAT では `vehicle_count_min`〜`vehicle_count_max` のレンジを設定し、各 k 台で独立した最適化を実行することで台数ごとの候補プランを生成していた。

Routing Library はコスト最小化によって時間内に数理的な最小台数を自動で求める。上限台数（`vehicle_count`）のみを設定すれば台数スイープは不要であり、ソルバーが時間内に最適な使用台数を決定する。このため `vehicle_count_min` による下限制約も不要となる。

この変更により depot_master の `vehicle_count_min` は削除済み。ソルバーへ渡すパラメータは上限台数（UI で指定した当日の稼働台数）のみとなる。

**複数候補プランの生成軸の変更**

台数スイープが不要になることで台数軸による候補プランの多様性は生まれない。そこで候補プランの生成軸を**初期解生成戦略（first_solution_strategy）**に変更する。上限台数を固定したまま異なる戦略で解を探索することで、異なるルート構造の候補プランを生成することを意図する。詳細は第 3 章で定義する。

**使用台数下限制約の廃止**

CP-SAT では `vehicle_count_min` による使用台数の下限強制が可能だった。Routing Library の API は上限制約のみを持つため、下限の強制は廃止される。評価項目は 11 項目から **9 項目**となる（「使用台数 ≥ 下限」の削除に加え、常に SKIP となる「距離精度 Google Maps 比較」も評価レポートから除外）。

## 2. 車両台数管理の設計変更

### 2.1 depot_master と UI の役割分担

**depot_master（恒久的なデポ情報として管理）**

デポの物理的・制度的な属性情報を管理する。滅多に変わらない情報を対象とする。

| 項目 | 理由 |
| --- | --- |
| デポ住所・緯度経度 | デポの物理的な場所。変更はデポ移転時のみ |
| `vehicle_count` | そのデポが保有する車両台数。保有台数が変わるのは稀。UI の初期値として使用される |
| `capacity_per_vehicle` | 1 台あたりの積載上限。車両の仕様に依存し、日々変わらない |
| 勤務時間・固定費・距離単価 | デポの運用ルール・コスト設計 |

`capacity_per_vehicle` は UI で設定する台数とは独立した「1 台ごとの物理制約」であるため、master に置くことが適切である。実際の総積載量は「UI 入力の稼働台数 × master の capacity_per_vehicle」で動的に計算される。

**UI（当日の実行パラメータ）**

| 項目 | 内容 |
| --- | --- |
| 今日の稼働台数 | 当日アサイン済みのドライバー数。初期値は depot_master の `vehicle_count` を表示する。当日の実際の稼働台数に応じて変更可能。depot_master の値を超える設定は不可 |

UI で入力した稼働台数が、ソルバーへの上限 k として渡される。depot_master の `vehicle_count` がバリデーション上限として機能する。

**バリデーション仕様**

- 入力値変更時：UI 入力値 > `vehicle_count` の場合、警告メッセージを表示する（実行はまだブロックしない）
- 実行ボタン押下時：UI 入力値 > `vehicle_count` の場合、エラーを表示して最適化を実行しない

### 2.2 数理最小台数のインサイト

Routing Library はコスト最小化を行うため、UI で稼働台数 k を指定しても実際の使用台数は k 以下になる場合がある。

```
例）稼働台数 10 を指定 → ソルバーは 8 台で全配送可能と判断 → 8 台を使用
```

この差分は「当日の稼働台数より少ない台数で配送が成立する」というインサイトであり、現場への示唆として有効である。UI の制約充足メトリクス（N/9 PASS）および評価レポートを確認することで使用台数の過不足を把握できる。

CP-SAT から Routing Library へ変更したことにより、`vehicle_count_min` による下限制約は廃止済み。depot_master のスキーマも更新済みであり、`vehicle_count_min` 列の削除・`vehicle_count_max` 列の `vehicle_count` への rename が完了している。

### 2.3 指定台数の強制使用に関する手法比較

本システムは前日に翌日の配送計画を作成することを想定している。当日にアサインされるドライバーの人数は前日時点で確定しているため、UI 入力台数は「翌日稼働するドライバー全員に仕事を割り当てる」という業務上の要件を表す。

指定台数を超えること（アサイン超過）も、下回ること（1 人以上が無配送）も現場運用上望ましくない。ちょうど指定台数のドライバー全員に配送計画を割り当てることが理想であるが、Routing Library の制約上これを直接強制することができない。

以下は将来対応の候補として考えられる手法とそのメリット・デメリットである。

| 手法 | 概要 | メリット | デメリット |
| --- | --- | --- | --- |
| **CP-SAT との併用** | 下限台数の強制が必要な台数のみ CP-SAT で解く | 制約を数学的に表現できる。最適性証明が可能な場合もある | 2 つのソルバーを維持する必要がある。大規模では CP-SAT がタイムアウトする可能性が残る。ライブラリ間で解の品質基準が異なる |
| **後処理によるルート分割** | Routing Library の解を取得後、最も長いルートを手動で分割する | 追加のソルバー不要。実装が比較的シンプル | 分割後のルートが時間帯制約・積載制約を満たす保証がない。ルートの品質が元の最適解より低下する |
| **間接制約の追加** | 1 台あたりの走行距離上限など別の制約を追加することで自然に台数を増やす | 単一ソルバーで解決できる | 本来の目的（台数確保）とは別の制約を追加するため設計の一貫性が失われる。意図しない副作用が生じる可能性がある |

いずれも一長一短がある。現 PoC ではいずれも未実装とし、数理最小台数をインサイトとして表示する運用を採用する。

## 3. 複数候補プランの生成方針変更

### 3.1 生成軸の変更

CP-SAT では台数（k 値）を軸に候補プランを生成していた。Routing Library では台数スイープを行わず上限台数を固定して解を探索するため、候補プランの生成軸を**初期解生成戦略（first_solution_strategy）**に変更する。

同一の目的関数（固定費 + 距離費）のまま初期解の構築方法を変えることで、異なるルート構造の候補プランを生成することを意図する。ただし、ヒューリスティクスの性質上、3 戦略が明確に異なるルート構造に収束することは保証されない。

| プラン | 初期解戦略 | 局所探索 | 特徴 |
| --- | --- | --- | --- |
| A | PATH_CHEAPEST_ARC | GUIDED_LOCAL_SEARCH | 近隣アーク優先のグリーディ解からスタート |
| B | SAVINGS | GUIDED_LOCAL_SEARCH | Clarke-Wright 法。台数統合によるコスト削減重視のスタート |
| C | CHRISTOFIDES | GUIDED_LOCAL_SEARCH | 理論的に高品質な初期解からスタート |

### 3.2 出力ファイルの変更

**route_summary.csv**

- `strategy` 列を追加（A / B / C）
- 各戦略の解を 1 行ずつ、計 3 行を格納（k は UI 入力の稼働台数で固定）
- 並び順は戦略 A→B→C で固定する（ソートしない）

**route_detail.csv**

- `strategy` 列を追加
- 戦略ごとの車両別ルートを格納（k は UI 入力の稼働台数で固定）

**可視化（ルートマップ）**

3 戦略それぞれの解に対してマップを生成する。

| ファイル名 | 用途 |
| --- | --- |
| `route_map_strategy_{A/B/C}.html` | 戦略ごとの全体俯瞰マップ |
| `route_map_vehicle_{id}_strategy_{A/B/C}.html` | 戦略ごとの車両別詳細マップ（outputs フォルダに保存） |

**UI マップタブ**

- 変更前：全体俯瞰タブ + 車両 1, 2, … タブ（ベストプランのみ）
- 変更後：戦略 A タブ + 戦略 B タブ + 戦略 C タブ（各戦略の全体俯瞰マップを表示）

## 5. OR-Tools Routing Library 技術仕様

### 5.1 OR-Tools Routing Library とは

OR-Tools Routing Library は Google が提供するオープンソース最適化ライブラリ OR-Tools に含まれる VRP 専用の API である。同パッケージの CP-SAT が汎用制約ソルバーであるのに対し、Routing Library は VRP の主要制約（全件訪問・デポ出発帰着・容量・時間窓・部分巡回防止）を組み込み機能として提供する。

ソルバーは以下の 2 フェーズで動作する。

1. **初期解生成**: `first_solution_strategy` に従い、最初のルート案を構築する
2. **局所探索による改善**: `local_search_metaheuristic` に従い、隣接解への移動を繰り返しコストを改善する。`time_limit` に達した時点で最良解を返す

ヒューリスティクスベースのため厳密な最適解の保証はないが、CP-SAT のような変数爆発が起きないため大規模問題に適している。

---

### 5.2 first_solution_strategy（初期解生成戦略）

最初のルートをどのように構築するかを決定する。探索の出発点となるため、最終解のルート構造の多様性に影響する。

```python
params.first_solution_strategy = routing_enums_pb2.FirstSolutionStrategy.PATH_CHEAPEST_ARC
```

| 定数名 | 特徴 |
| --- | --- |
| `AUTOMATIC` | 問題の構造に応じてソルバーが自動選択する。バージョンや問題特性により選ばれる戦略は異なる |
| `PATH_CHEAPEST_ARC` | デポから出発し、最もコストの低い隣接ノードを次々につなぐ。高速だが全体バランスを欠く場合がある |
| `SAVINGS` | Clarke-Wright 法。全地点を個別往復する場合と比較し、ルートを統合した際のコスト削減量が大きい順に結合する。台数統合の観点から効率的な初期解を生成しやすい |
| `CHRISTOFIDES` | 最小全域木をベースとするアルゴリズム。理論的に最適解の 1.5 倍以内の初期解を生成することが証明されている |
| `PARALLEL_CHEAPEST_INSERTION` | 複数のルートに並行して、最もコスト増が少ないノードを挿入していく |
| `GLOBAL_CHEAPEST_INSERTION` | 全ルート・全ノードの中で挿入コストが最も低い場所へ一つずつ配置する |
| `LOCAL_CHEAPEST_INSERTION` | 現在構築中のルートに対して最適な挿入場所を探す |
| `FIRST_UNBOUND_MIN_VALUE` | 制約プログラミングの標準的な探索手法。VRP では一般的に他の戦略より劣る |

---

### 5.3 local_search_metaheuristic（局所探索メタヒューリスティクス）

初期解を改善し続けるための戦略。局所最適解（改善できない行き止まり）からの脱出能力が解の品質を左右する。

```python
params.local_search_metaheuristic = routing_enums_pb2.LocalSearchMetaheuristic.GUIDED_LOCAL_SEARCH
```

| 定数名 | 特徴 |
| --- | --- |
| `AUTOMATIC` | OR-Tools が問題の構造に応じて戦略を選択する |
| `GREEDY_DESCENT` | コストが改善する操作のみを受け入れる。高速だが局所最適解に陥りやすい。小規模・シンプルな問題向き |
| `GUIDED_LOCAL_SEARCH` | 頻出アークにペナルティを課し、強制的に別のルートを探索させる。VRP・時間窓制約のある問題で最も推奨される手法 |
| `SIMULATED_ANNEALING` | 焼きなまし法。探索初期は改悪も確率的に許容し、徐々に厳しくなる。多様な解が得やすく、問題によっては GLS より良い解が出ることがある |
| `TABU_SEARCH` | 直近の操作を一定期間禁止し、同じ解への再訪を防ぐ。探索空間を広く探索できるが、パラメータ調整が必要 |

---

### 5.4 time_limit（探索時間上限）

探索時間の上限（秒）。この時間に達した時点でそれまでの最良解を返す。時間内に実行可能解が見つからない場合は `None` を返す。

```python
params.time_limit.seconds = 120
```

ヒューリスティクス系ソルバーは探索開始直後に大きな改善が得られ、時間が経つにつれ改善幅が小さくなる傾向がある（収束）。そのため時間を無制限に延ばしても解の品質が比例して上がるわけではない。

| 設定値 | 想定される挙動 |
| --- | --- |
| **30 秒** | 小規模データ（〜20 配送先、k≤5）に対しては収束に十分な時間。繰り返し実行しても比較的安定した解が得られやすい。100 配送先規模では実行可能解が見つからない可能性がある |
| **60 秒** | 中規模データ（〜50 配送先、k≤10）に対して有効。解の品質はある程度安定するが、実行ごとの差異が生じ始める |
| **120 秒** | 大規模データ（100 配送先、k≤12）に対して現実的な解が得られることが多い。本 PoC のデフォルト設定。実行ごとに解が異なる可能性がある |
| **300 秒** | 複雑な時間帯制約や大規模データに対して探索の安定性が高まる。ただし 120 秒以降は改善幅が小さくなることが多く、実運用での応答性とのトレードオフを考慮する必要がある |

---

### 5.5 探索ログについての注意

`log_search_progress` は CP-SAT（`SatParameters`）専用のパラメータであり、Routing Library の `RoutingSearchParameters` には存在しない。設定しようとすると `AttributeError` が発生する。

Routing Library の探索ログを出力したい場合は OR-Tools v9.x 以降で提供される別のログ制御機能（環境変数等）を利用する必要があるが、本 PoC では実装していない。
