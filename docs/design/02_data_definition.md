# データ定義

このドキュメントでは、VRP PoC におけるデータ定義を整理する。

## 1. ドキュメントの目的

本ドキュメントの目的は、`00_problem_definition.md` および `01_business_requirements.md` で整理した問題設定と業務要件をもとに、本 PoC で扱うデータを論理的に定義することである。

本ドキュメントでは、PoC の実行に必要なデータを、どのような役割で保持し、どこから受け取り、どのような単位で扱うかを明確にする。あわせて、マスタデータ、トランザクションデータ、前処理生成データ、出力データの区分を整理し、後続のネットワーク設計、数理モデル設計、実装設計へ接続できる状態を目指す。

特に本 PoC では、単にカラム一覧を列挙するのではなく、業務上必要なデータと内部処理で生成されるデータを切り分け、再現可能かつ説明可能なデータ構成を設計することを重視する。

そのため、本ドキュメントでは以下を整理対象とする。

- データ定義の基本方針
- データの提供元
- 論理データ構成
- マスタデータ、トランザクションデータ、前処理生成データ、出力データの定義
- データ間の関係
- 更新および保存の方針

これにより、本 PoC を構成するデータの役割と関係を明確にし、後続の設計と実装で迷いなく参照できる状態を作る。

## 2. データ定義の方針

本 PoC では、配送計画に必要なデータを役割ごとに分けて整理する。  
対象とするデータ区分は、マスタデータ、トランザクションデータ、前処理生成データ、OSM マップデータ、出力データの 5 つとする。

業務上の入力データとして扱うのは、デポ条件や車両条件を保持するマスタデータと、配送日ごとに変動する配送トランザクションデータである。  
一方で、緯度経度、住所正規化結果、道路ネットワーク割当結果などは、入力時点では存在せず、PoC 側の前処理によって生成される内部データとして扱う。

本 PoC におけるデータの基本的な流れは、配送データとして受け取った住所情報と荷物数をもとに、住所前処理によって GIO 情報を付与し、その結果を OSM マップデータへ割り当てたうえで、距離・時間計算および最適化へ利用するという構成を前提とする。  
そのため、業務入力としてのデータと、最適化処理のために内部で生成・参照するデータを明確に分けて定義する。

本ドキュメントでは、これらのデータを論理的に定義することを目的とし、どのデータをどの単位で持つべきか、どのような役割で利用するか、どのような関係を持つかを整理対象とする。  
一方で、物理的な保存方式、利用するデータベース製品、クラウド構成といった実装レベルの詳細は、本ドキュメントの対象外とする。

また、本 PoC では再現性を重視するため、同一の入力データ、同一の OSM マップデータ、同一の前処理条件、同一の計算条件を用いた場合に、同一の結果を再生成できるデータ構成を目指す。  
そのため、入力データだけでなく、前処理によって生成されるデータや、実行時に参照する基盤データについても定義対象に含める。

これにより、本ドキュメントで整理したデータ定義を、後続の道路ネットワーク設計、数理モデル設計、実装設計へ接続できる状態を作る。

## 3. データ提供元

本 PoC で扱うデータは、外部から取得する基盤データ、PoC 用に定義する内部マスタ、外部入力または疑似生成によって用意する配送データ、前処理基盤によって生成されるデータに分けて扱う。  
現時点では実クライアントの配送実績データを保有していないため、配送トランザクションデータは、公開可能な入力データまたは PoC 用に作成した疑似データを利用する前提とする。

| データ区分 | 提供元 | 用途 | 補足 |
|---|---|---|---|
| OSM マップデータ | OpenStreetMap | 道路ネットワークの正本として利用する | 対象都道府県および配送対象エリアの道路グラフを取得し、ローカル保存して再利用する |
| デポ・車両条件マスタ | PoC 内部定義データ | デポ住所、台数レンジ、固定費、車両容量、業務時間などを保持する | 実クライアントマスタではなく、PoC 用に定義する内部マスタとして扱う |
| 配送先マスタ | PoC 内部定義データ | 配送先識別子、配送先名称、住所情報を保持する | 新規配送先入力時に配送先単位の安定識別子を付与するために利用する |
| 配送トランザクションデータ | 外部入力データまたは PoC 用疑似データ | 配送日、配送先住所、荷物数、時間帯区分を保持する | 現時点では実クライアントデータを持たないため、公開可能な入力データ、手動作成データ、または生成 AI 等を用いた疑似データで代替する |
| 住所前処理データ | `geo_enrichment_streamlit` | 住所正規化、ジオコーディング、緯度経度付与を行う | 配送データとデポデータの前処理結果として利用する |
| 出力データ | VRP PoC 内部生成データ | 候補計画サマリ、ルート明細、ルート可視化成果物を生成する | 最適化処理の結果として生成される |

## 4. 論理データ構成

本 PoC では、配送計画に必要なデータを、役割ごとに複数の論理データ群へ分けて扱う。  
ここでいう論理データ構成とは、データをどのようなまとまりで保持し、誰が提供し、誰が利用し、どのタイミングで更新され、どの結果を次工程へ渡すかを整理したものである。

本 PoC における論理データ構成は、以下の 5 つを基本とする。

| 論理データ群 | 主な内容 | 提供者 | 主な利用者 | 更新タイミング | 役割 | 後続での利用先 |
|---|---|---|---|---|---|---|
| マスタデータ | デポ情報、配送先識別情報、台数レンジ、固定費、車両容量、業務時間など | PoC 管理者 | 配送計画担当者、VRP PoC | 条件変更時 | 配送計画の前提条件および配送先の安定識別情報を保持する | 前処理、最適化、出力生成 |
| トランザクションデータ | 配送日、配送先住所、荷物数、時間帯区分など | 外部入力元または PoC 用疑似データ作成者 | 配送計画担当者、VRP PoC | 配送日ごと | 日次で変動する配送対象データを保持する | 前処理、最適化、出力生成 |
| 前処理生成データ | 正規化後住所、緯度経度、道路ネットワーク割当結果など | `geo_enrichment_streamlit` および VRP PoC | VRP PoC | 実行時または再処理時 | 業務入力データを最適化可能な形式へ変換する | 距離・時間計算、最適化 |
| OSM マップデータ | 道路ネットワーク、ノード、エッジ、道路属性など | OpenStreetMap | VRP PoC | OSM 更新時または基盤更新時 | 実道路距離およびルート計算の基盤となる | 配送地点割当、距離・時間計算、可視化 |
| 出力データ | 候補計画サマリ、ルート明細、ルート可視化成果物など | VRP PoC | 配送計画担当者、実配送担当者、導入判断者 | 実行ごと | 最適化結果を業務利用可能な形式で返す | 業務判断、現場確認、PoC 評価 |

これらのデータ群は独立して存在するのではなく、配送トランザクションデータとマスタデータを起点として、前処理生成データを介して OSM マップデータと接続し、最終的に出力データを生成する流れを前提とする。  
そのため、本ドキュメントでは各データ群を個別に定義するだけでなく、後続章でそれぞれの関係も整理する。

## 5. マスタデータ定義

本 PoC では、マスタデータとして単一デポに紐づく配送計画の前提条件を保持する。  
ここで定義するマスタデータは、日々変動する配送トランザクションデータとは異なり、条件変更時にのみ更新される基準情報として扱う。

現時点では、マスタデータはデポ情報を中心に構成し、台数レンジ、コスト、車両容量、業務時間といった最適化の前提条件に加え、到着予定区分や配送先識別のようなコード管理が望ましい項目も保持する。  
デポについては、住所、緯度経度、道路ネットワーク割当結果をマスタ側で保持し、未設定項目がある場合にのみ前処理および割当を実行する前提とする。

|---|---|---|---|---|---|---|
| 時間帯区分コード | time_slot_code | 到着予定区分を識別するコード | integer | 必須 | PK | 区分値を一意に識別する |
| 時間帯区分名 | time_slot_name | 到着予定区分の名称 | string | 必須 | なし | `午前 / 午後 / 時間指定なし` を想定する |
| 開始時刻 | slot_start_time | 当該区分の開始時刻 | time | 任意 | なし | 例: 午前 = 09:00 |
| 終了時刻 | slot_end_time | 当該区分の終了時刻 | time | 任意 | なし | 例: 午前 = 13:00 |

本 PoC では、到着予定区分をコード値として管理し、`1 = 午前`、`2 = 午後`、`3 = 時間指定なし` を基本値として扱う。  
`時間指定なし` は明示的な時間帯制約を持たないため、開始時刻と終了時刻は任意項目として扱う。

| 論理名 | 物理名 | 説明 | データ型 | 必須 | キー種別 | 備考 |
|---|---|---|---|---|---|---|
| 配送ID | delivery_id | 配送先を識別する ID | string | 必須 | PK | UUID により採番する |
| 配送先名 | destination_name | 配送先の名称 | string | 任意 | なし | 表示用名称 |
| 配送先住所 | destination_address | 配送先所在地の元住所情報 | string | 必須 | なし | 入力時点の住所を保持する |
| 正規化後住所 | normalized_address | 同一配送先判定に用いる正規化後住所 | string | 任意 | なし | 前処理結果を反映する |
| 緯度 | latitude | 配送先マスタで保持する配送先の緯度 | float | 任意 | なし | 新規配送先登録時に付与し、既存配送先では再利用する |
| 経度 | longitude | 配送先マスタで保持する配送先の経度 | float | 任意 | なし | 新規配送先登録時に付与し、既存配送先では再利用する |
| 登録日時 | created_at | 配送先マスタへ登録した日時 | datetime | 必須 | なし | 新規配送先登録時に記録する |
| 更新日時 | updated_at | 配送先マスタの最終更新日時 | datetime | 任意 | なし | 住所修正時などに更新する |

本 PoC では、`delivery_id` を配送先単位の安定識別子として扱う。  
本配送先マスタは、外部の正式マスタを受領する前提ではなく、配送トランザクションを起点として PoC 内部で生成・維持する内部生成型マスタとして扱う。  
新規配送先が入力された場合は、正規化後住所をもとに既存配送先を照合し、一致しない場合は新規配送先として配送先マスタへ登録し、UUID を付与する。  
その際、新規配送先では前処理により取得した緯度経度もあわせて配送先マスタへ保存する。  
既存配送先については、配送先マスタに保存済みの緯度経度を再利用し、同一配送先に対する位置のぶれを抑える。  
その後、採番済みの `delivery_id` を `delivery_transaction` へ紐づける。

## 6. トランザクションデータ定義

本 PoC では、トランザクションデータとして配送日ごとに変動する配送対象データを保持する。  
ここで定義するトランザクションデータは、配送計画実行時の業務入力となるものであり、配送日、配送先、荷物数、時間帯区分などを管理する。

本 PoC では、配送先そのものを識別する ID と、配送レコードを識別する ID を分けて扱う。  
これにより、同一配送先に対して複数回の配送が発生する場合や、配送日単位でのレコード管理が必要な場合にも、論理的に切り分けて扱える構成とする。

## 7. 前処理生成データ定義

本 PoC では、入力時点で保持している業務データを、そのまま最適化に利用するのではなく、住所前処理、道路ネットワーク割当、距離・時間計算を通じて、最適化可能な形式へ変換する。  
そのため、本章では前処理によって生成されるデータを定義する。

前処理生成データは、大きく、住所前処理結果、配送地点割当結果、距離・時間計算用データの 3 つに分けて扱う。  
なお、OSM のノード・エッジ属性そのものは前処理生成データではなく、道路ネットワーク基盤として別管理する。本章では、それらを利用して生成される参照結果や計算結果を対象とする。  
また、配送先については新規配送先登録時に住所前処理結果を配送先マスタへ反映し、既存配送先では配送先マスタに保存済みの緯度経度を再利用する前提とする。

| 経度 | longitude | ジオコーディングにより付与した経度 | float | 任意 | なし | 変換成功時のみ保持する |
| ジオコーディング成否 | geocode_status | 緯度経度変換の成否 | string | 必須 | なし | 例: success / failed |
| ジオコーディング元 | geocode_source | 使用した前処理基盤または変換元 | string | 任意 | なし | 本 PoC では `geo_enrichment_streamlit` を想定する |

| 計画ID | plan_id | 距離・時間計算対象となる配送計画実行単位の ID | string | 必須 | なし | 1 回の PoC 実行単位で付与する |

| 到着ノードID | destination_node_id | 到着地点の道路ネットワークノード ID | string | 必須 | なし | デポまたは配送先を想定する |
| 道のり距離 | path_distance | OSM ネットワーク上の道のり距離 | float | 必須 | なし | ルート評価とコスト算出に利用する |
| 静的推定時間 | estimated_travel_time | OSM 道路属性を用いた静的推定所要時間 | float | 任意 | なし | 時間制約判定に利用する |
| 時間補正係数 | travel_time_adjustment_factor | 所要時間補正に用いる係数 | float | 任意 | なし | 補正ロジックは後続設計で定義する |

## 8. 出力データ定義

本 PoC では、最適化処理の結果を単一の確定計画として自動決定するのではなく、配送計画担当者が比較・判断できる候補案群として出力する。  
そのため、出力データは「採用済み計画」を表すデータではなく、システムが生成した候補案一覧、各候補案のルート明細、候補案のルート形状を確認するための可視化成果物を中心に定義する。

また、本 PoC では最終採用判断は人手で行う前提とする。  
一方で、システムとして何を最良候補と判定したかを利用者へ明示することは、比較負荷の軽減と説明可能性の観点で有効である。  
そのため、各候補案には順位情報に加えて、システム上の推奨案であることを示す情報を持たせる。

### 8.1 candidate_plan_summary

`candidate_plan_summary` は、1 回の PoC 実行で生成された候補計画を、案単位で比較するための出力データである。  
配送計画担当者および導入判断者が、使用台数、総走行距離、総概算コスト、制約充足状況を一覧で比較し、確認対象とする案を選べるようにするために利用する。

| 論理名 | 物理名 | 説明 | データ型 | 必須 | キー種別 | 備考 |
|---|---|---|---|---|---|---|
| 候補計画ID | candidate_plan_id | 候補案を識別する ID | string | 必須 | PK | 案ごとに一意な値 |
| 計画ID | plan_id | 同一実行に属する候補案群を識別する ID | string | 必須 | なし | 1 回の PoC 実行単位 |
| 配送日 | delivery_date | 対象配送日 | date | 必須 | なし | 日次計画単位 |
| デポID | depot_id | 対象デポ ID | string | 必須 | FK | `depot_master.depot_id` を参照する |
| 使用台数 | vehicle_count | 当該案の使用台数 | integer | 必須 | なし | 台数が異なれば別案として扱う |
| 総走行距離 | total_distance | 当該案の総走行距離 | float | 必須 | なし | 距離比較用 |
| 総概算コスト | total_cost | 当該案の概算コスト | float | 必須 | なし | `固定費 + 距離比例費` |
| 固定費部分 | fixed_cost_component | 当該案の固定費部分 | float | 必須 | なし | `車両固定費 × 使用台数` |
| 距離比例費部分 | distance_cost_component | 当該案の距離比例費部分 | float | 必須 | なし | `距離単価 × 総走行距離` |
| 総訪問件数 | total_stop_count | 当該案に含まれる訪問件数 | integer | 必須 | なし | 全件訪問確認用 |
| 総荷物個数 | total_package_count | 当該案で配送する荷物総数 | integer | 必須 | なし | 全量配送確認用 |
| 業務時間内帰着可否 | return_within_work_hours | 業務時間内帰着条件を満たすか | boolean | 必須 | なし | `1 日 8 時間以内帰着` の確認用 |
| 時間帯制約充足可否 | time_window_feasible | 時間帯制約を満たすか | boolean | 必須 | なし | `午前 / 午後 / 時間指定なし` の確認用 |
| 実行可能解フラグ | feasible_flag | 制約を満たす実行可能解か | boolean | 必須 | なし | 案の採用候補可否を表す |
| 全件配送フラグ | all_deliveries_included | 指定された配送先全件が対象に含まれているか | boolean | 必須 | なし | `false` の場合は一部配送先を除外した暫定結果であることを示す |
| 評価順位 | rank | システム評価上の順位 | integer | 必須 | なし | `1` が最上位候補 |
| システム推奨フラグ | is_system_recommended | システム上の推奨案かどうか | boolean | 必須 | なし | 一覧上で強調表示する対象 |
| 推奨理由 | recommendation_reason | 推奨案とした理由の要約 | string | 任意 | なし | 例: `最少台数かつ総コスト最小` |
| 目的関数値 | objective_value | システム内部評価に用いた目的関数値 | float | 任意 | なし | 順位根拠の保持用 |
| 生成日時 | generated_at | 候補案の生成日時 | datetime | 必須 | なし | 実行履歴確認用 |

本 PoC では、`rank = 1` の案を原則として `is_system_recommended = true` として扱う。  
ただし、最終採用判断は配送計画担当者が行うため、この推奨情報はあくまで比較起点を示す補助情報として扱う。

また、順位付けの対象とするのは `feasible_flag = true` を満たす候補案に限定する。  
順位は、まず `vehicle_count` が少ない案を優先し、同一台数内では `total_cost` が小さい案を上位とし、それでも同値の場合は `total_distance` が短い案を上位とする。  
この順位付けにおいて `rank = 1` となった案を、システム上の推奨案として `is_system_recommended = true` とする。  
ここでの `rank` は、候補案を業務判断しやすい順に提示するための順位であり、ソルバー内部で用いる目的関数値そのものとは役割を分けて扱う。

### 8.2 candidate_route_detail

`candidate_route_detail` は、各候補計画に含まれるルートの詳細を、配送先単位で確認するための明細データである。  
配送計画担当者および実配送担当者が、各案の巡回順序、配送先構成、区間距離、累積距離、時間帯想定を具体的に確認できるようにするために利用する。

| 論理名 | 物理名 | 説明 | データ型 | 必須 | キー種別 | 備考 |
|---|---|---|---|---|---|---|
| 計画ID | plan_id | 対象計画実行 ID | string | 必須 | なし | 1 回の PoC 実行単位 |
| 候補計画ID | candidate_plan_id | 対象候補案 ID | string | 必須 | FK | `candidate_plan_summary.candidate_plan_id` を参照する |
| ルートID | route_id | 車両単位のルートを識別する ID | string | 必須 | なし | 候補案内で一意な値 |
| 車両番号 | vehicle_no | 表示用の車両番号 | integer | 必須 | なし | `1, 2, 3...` を想定する |
| 訪問順序 | stop_order | 当該ルート内での訪問順序 | integer | 必須 | なし | デポ出発後の順序を表す |
| トランザクションID | transaction_id | 元となる配送レコード ID | string | 必須 | FK | `delivery_transaction.transaction_id` を参照する |
| 配送ID | delivery_id | 対象配送先を識別する ID | string | 必須 | FK | `delivery_destination_master.delivery_id` を参照する |

| 荷物個数 | package_count | 当該配送先の荷物個数 | integer | 必須 | なし | 需要量として扱う |
| 累積荷物個数 | cumulative_package_count | 当該車両で当該配送先までに積載対象として扱う累積荷物個数 | integer | 任意 | なし | 積載状況確認用 |
| 指定時間帯区分 | requested_time_slot | 指定された到着予定区分 | string | 必須 | なし | `午前 / 午後 / 時間指定なし` |

これにより、候補案比較とルート内容確認を同一のデータ構造上で接続できるようにする。

### 8.3 visualization_artifact

`visualization_artifact` は、候補案ごとのルート案を地図上で確認するための可視化成果物の管理データである。  
表形式データだけでは把握しにくい巡回順序やルート形状を、配送計画担当者および実配送担当者が視覚的に確認できるようにするために利用する。

| 成果物ID | artifact_id | 可視化成果物を識別する ID | string | 必須 | PK | ファイル単位で一意な値 |
| 計画ID | plan_id | 同一実行に属する成果物群を識別する ID | string | 必須 | なし | 1 回の PoC 実行単位 |
| 候補計画ID | candidate_plan_id | 対象候補案 ID | string | 必須 | FK | `candidate_plan_summary.candidate_plan_id` を参照する |
| 成果物種別 | artifact_type | ルート可視化成果物の種別 | string | 必須 | なし | `overall_route_map / route_map / stop_order_route_map` を想定する |

本 PoC では、可視化成果物として少なくとも以下を出力対象とする。

- 全体ルート図  
  候補案全体のルート構成を俯瞰できる可視化。デポ位置、配送先位置、ルートごとの色分けを含む。
- ルート別可視化  
  各車両ルートごとの巡回経路を確認できる可視化。ルート単位で拡大し、現場が確認しやすい粒度で表示する。
- 訪問順序付き可視化  
  配送先ごとに `1, 2, 3...` の訪問順を付番して表示する可視化。配送担当者が巡回順序を直感的に理解できるようにする。

本 PoC では、出力データの中心を `candidate_plan_summary` と `candidate_route_detail` に置く。  
`candidate_plan_summary` は候補案比較のための集約データ、`candidate_route_detail` は各候補案の具体的な巡回内容を確認するための明細データ、`visualization_artifact` はルート案を視覚的に確認するための補助データとして扱う。  
また、最終採用判断は人手で行う前提であるため、出力データには確定計画を表す独立データは持たず、システム推奨案を補助情報として提示する構成とする。

## 9. データ間の関係

本章では、前章までに定義したマスタデータ、トランザクションデータ、前処理生成データ、出力データの関係を整理する。  
ここで整理するのは、各データがどの ID を介して紐づくか、どの単位で親子関係を持つか、どの粒度で管理されるかという、いわば ER 図に相当する論理関係である。

本 PoC では、データの関係を理解するうえで、以下の 4 つの粒度を区別することが重要である。

- `transaction_id`  
  配送レコード単位を識別する ID
- `delivery_id`  
  配送先単位を識別する安定 ID
- `plan_id`  
  1 回の配送計画実行単位を識別する ID
- `candidate_plan_id`  
  1 回の配送計画実行の中で生成される候補案単位を識別する ID

この 4 つの粒度を明確に分けることで、配送先、配送レコード、計画実行、候補案、ルート明細、可視化成果物の関係を一貫して扱えるようにする。

### 9.1 マスタデータとトランザクションデータの関係

マスタデータとトランザクションデータの関係は、単一デポ前提の配送計画を構成する基礎関係である。  
`delivery_transaction` は配送日ごとに変動する入力データであり、その前提条件を `depot_master`、`time_slot_master`、`delivery_destination_master` が支える構造とする。

関係は以下のとおりとする。

- `delivery_transaction.depot_id` は `depot_master.depot_id` を参照する
- `delivery_transaction.delivery_time_slot_code` は `time_slot_master.time_slot_code` を参照する
- `delivery_transaction.delivery_id` は `delivery_destination_master.delivery_id` を参照する

関係の多重度は以下のとおりである。

- 1 件の `depot_master` に対して、複数件の `delivery_transaction` が紐づく
- 1 件の `time_slot_master` に対して、複数件の `delivery_transaction` が紐づく
- 1 件の `delivery_destination_master` に対して、複数件の `delivery_transaction` が紐づきうる

本 PoC は単一デポ前提であるが、論理設計上は `delivery_transaction` 側に `depot_id` を保持することで、将来的な複数デポ拡張にも耐えられる形とする。

### 9.2 トランザクションデータと前処理生成データの関係

前処理生成データは、業務入力として受け取った配送トランザクションを、最適化可能な形式へ変換した結果を保持する。  
このときの親データは `delivery_transaction` であり、前処理生成データは `transaction_id` を介して元レコードへ紐づく。

関係は以下のとおりとする。

- `geocode_result.transaction_id` は `delivery_transaction.transaction_id` を参照する
- `network_assignment_result.transaction_id` は `delivery_transaction.transaction_id` を参照する

関係の多重度は、基本的に以下を想定する。

- 1 件の `delivery_transaction` に対して、1 件の `geocode_result`
- 1 件の `delivery_transaction` に対して、1 件の `network_assignment_result`

ただし、再処理や履歴管理を物理設計で扱う場合には、同一 `transaction_id` に対して複数履歴を持つ可能性がある。  
本ドキュメントでは論理設計として、各配送レコードに対して最新または採用済みの前処理結果が 1 件紐づく前提で扱う。

### 9.3 前処理生成データと距離・時間計算データの関係

`od_matrix_result` は、配送計画実行単位で必要となる地点間距離・所要時間を保持するデータである。  
このデータは、`network_assignment_result` によって付与された道路ネットワーク上の参照地点をもとに生成される。

関係の考え方は以下のとおりである。

- 配送先については `network_assignment_result.network_node_id` をもとに、距離・時間計算対象のノード集合を構成する
- デポについては `depot_master.depot_network_node_id` を出発地点および帰着地点のノードとして利用する
- そのノード集合に対して、`plan_id` 単位で `od_matrix_result` を生成する

ここで重要なのは、`od_matrix_result` が `transaction_id` 単位ではなく `plan_id` 単位で管理される点である。  
すなわち、個々の配送レコードに紐づく地点情報を入力としつつ、距離・時間計算結果そのものは「その日の配送計画実行に必要な OD 集合」として保持する。

### 9.4 計画実行単位と候補案データの関係

出力データでは、`plan_id` を配送計画実行単位、`candidate_plan_id` をその実行内で生成される候補案単位として扱う。  
この関係により、1 回の実行で複数候補案を保持しつつ、それらが同一条件下で生成されたことを明確にできる。

関係は以下のとおりとする。

- 1 件の `plan_id` に対して、複数件の `candidate_plan_summary` が紐づく
- `candidate_plan_summary.plan_id` は、同一実行に属する候補案群を識別する

関係の多重度は以下のとおりである。

- 1 件の `plan_id` に対して、複数件の `candidate_plan_summary`
- 1 件の `candidate_plan_summary` は、必ず 1 件の `plan_id` に属する

この構造により、同一入力条件・同一実行条件のもとで生成された候補案群を、同一の計画実行に属するものとしてまとめて扱えるようにする。

### 9.5 候補案データとルート明細データの関係

`candidate_route_detail` は、各候補案に含まれるルートの詳細を保持する子データである。  
候補案の比較は `candidate_plan_summary` で行い、各候補案の具体的な巡回内容は `candidate_route_detail` で確認する構成とする。

関係は以下のとおりとする。

- `candidate_route_detail.candidate_plan_id` は `candidate_plan_summary.candidate_plan_id` を参照する
- `candidate_route_detail.plan_id` は `candidate_plan_summary.plan_id` と同一の計画実行単位を表す
- `candidate_route_detail.transaction_id` は `delivery_transaction.transaction_id` を参照する
- `candidate_route_detail.delivery_id` は `delivery_transaction.delivery_id` を参照する

関係の多重度は以下のとおりである。

- 1 件の `candidate_plan_summary` に対して、複数件の `candidate_route_detail`
- 1 件の `candidate_route_detail` は、必ず 1 件の `candidate_plan_summary` に属する
- 1 件の `delivery_transaction` は、候補案ごとに異なる `candidate_route_detail` へ現れうる

最後の点が重要であり、同一配送先は複数候補案の中で異なる車両、異なる訪問順序に割り当てられる可能性がある。  
そのため、`candidate_route_detail` は配送レコードの複製ではなく、「候補案における配送レコードの配置結果」を表すデータとして扱う。

### 9.6 候補案データと可視化成果物の関係

`visualization_artifact` は、候補案ごとのルート案を視覚的に確認するための成果物管理データである。  
表形式データとは別に、候補案に対応する地図画像やルート図を管理するために利用する。

関係は以下のとおりとする。

- `visualization_artifact.plan_id` は計画実行単位を表す
- `visualization_artifact.candidate_plan_id` は対象候補案を表す

関係の多重度は以下のとおりである。

- 1 件の `candidate_plan_summary` に対して、複数件の `visualization_artifact`
- 1 件の `visualization_artifact` は、必ず 1 件の `candidate_plan_summary` に属する

本 PoC では、1 つの候補案に対して少なくとも以下の複数成果物が紐づきうる。

- 全体ルート図
- ルート別可視化
- 訪問順序付き可視化

### 9.7 本 PoC における主要な親子関係

本 PoC における主要な親子関係をまとめると、以下のとおりである。

- `depot_master` 1 : N `delivery_transaction`
- `time_slot_master` 1 : N `delivery_transaction`
- `delivery_destination_master` 1 : N `delivery_transaction`
- `delivery_transaction` 1 : 1 `geocode_result`
- `delivery_transaction` 1 : 1 `network_assignment_result`
- `plan_id` 1 : N `od_matrix_result`
- `plan_id` 1 : N `candidate_plan_summary`
- `candidate_plan_summary` 1 : N `candidate_route_detail`
- `candidate_plan_summary` 1 : N `visualization_artifact`

### 9.8 関係整理上の補足

本 PoC では、データ間の関係を理解するうえで、以下を前提とする。

- `transaction_id` は配送レコード単位の識別子である
- `delivery_id` は配送先単位の安定識別子である
- `plan_id` は 1 回の配送計画実行単位の識別子である
- `candidate_plan_id` は 1 回の配送計画実行の中で生成される候補案単位の識別子である

この 4 層の識別子を分けることで、入力データ、前処理結果、距離・時間計算、候補案比較、ルート明細、可視化成果物を一貫した構造で接続できるようにする。

## 10. 更新方針

本章では、本 PoC で扱う各データの更新に関する基本方針を整理する。  
ここでいう更新とは、単に元データを差し替えることだけでなく、関連する生成データや出力データを再作成することも含む。

### 10.1 更新方針の基本的な考え方

本 PoC では、入力データ、前処理生成データ、最適化結果、可視化成果物について、それぞれ更新契機に応じた再生成を行う。  
また、本 PoC では再現性を重視するため、毎回すべての基盤データを無条件に更新するのではなく、更新対象ごとに更新契機を定めたうえで、必要な範囲のみ再生成する方針とする。

### 10.2 OSMデータ

OSMデータは、道路ネットワークの正本データとして利用する。  
更新頻度については、OSM 側の更新特性を踏まえ、月次で更新要否を確認し、必要と判断した場合に正本データを更新する運用とする。

PoC段階では、毎回最新データを再取得するのではなく、利用するOSMデータの版を一定期間固定することで、結果の再現性を確保する。  
そのうえで、道路事情の変化を過度に取りこぼさないよう、1か月単位で見直す。

### 10.3 マスタデータ

デポ情報や時間帯区分などのマスタデータは、業務条件の変更時に更新する。  
通常は高頻度で変化しない前提とし、定期更新ではなくイベント駆動での更新を基本とする。

更新対象の例は以下とする。

- デポ位置やデポ名称の変更
- 時間帯区分の追加、変更、廃止
- コスト算出条件に関わる基準情報の変更
- 新規配送先入力時の配送先マスタ追加
- 配送先名称や配送先住所の修正に伴う配送先マスタ更新

### 10.4 配送トランザクションデータ

配送トランザクションデータは、配送日ごとの入力データとして扱う。  
配送先、住所、荷物個数、指定時間帯などの情報に変更があった場合は、対象配送日のデータを更新する。

実運用上は、配送日単位で新規受領または差し替えが発生するため、実質的には日次更新を基本とする。

### 10.5 前処理生成データ

ジオコーディング結果および道路ネットワーク割当結果などの前処理生成データは、配送トランザクションデータをもとに生成する。  
そのため、配送トランザクションデータに変更があった場合は、対応する前処理生成データを再生成する。

また、前処理生成データは入力データ変更時だけでなく、以下の場合にも再生成対象とする。

- OSMデータを更新した場合
- ジオコーディング処理ロジックを変更した場合
- ネットワーク割当条件を変更した場合

### 10.6 距離・時間計算データ

距離・時間計算データである `od_matrix_result` は、配送計画実行単位である `plan_id` ごとに生成する。  
配送対象集合または道路ネットワーク条件が変わった場合は、対象 `plan_id` の距離・時間計算データを再生成する。

再生成契機は主に以下とする。

- 配送トランザクションデータの変更
- 前処理生成データの変更
- OSMデータの更新

### 10.7 候補案出力データ

候補案出力データは、VRP実行時に `plan_id` 単位で生成する。  
そのため、配送条件、コスト条件、探索条件、制約条件のいずれかが変化した場合は、対象 `plan_id` の候補案出力データを再生成する。

再生成契機は主に以下とする。

- 配送トランザクションデータの変更
- 車両台数レンジの変更
- コスト係数の変更
- 制約条件の変更
- VRPロジックまたは探索条件の変更

### 10.8 可視化成果物

可視化成果物は、候補案出力データに対応して生成する補助成果物である。  
そのため、可視化成果物単独で更新するのではなく、候補案出力データの再生成時にあわせて再生成する。

対象とする成果物は以下とする。

- 全体ルート図
- ルート別可視化
- 訪問順序付き可視化

## 11. 保存方針

本章では、本 PoC で扱う各データの保存に関する基本方針を整理する。  
保存の目的は、再現性の確保、実行履歴の比較、実行後の後追い確認、エラー発生時の原因調査である。  
そのため、本 PoC では最新結果のみを上書き保持するのではなく、配送計画実行単位ごとに結果を保存する方針とする。

### 11.1 保存方針の基本的な考え方

本 PoC では、保存単位を `plan_id` 単位とする。  
`plan_id` は 1 回の配送計画実行単位を表す識別子であり、実行日時と対象配送日を識別できる命名規則とする。

命名規則は、実行日時と対象配送日を明確に区別できるよう、以下の形式を基本とする。

- `YYYYMMDD_HHMMSS_YYYYMMDD`

前半は実行日時、後半は対象配送日を表す。  
これにより、どの時点で実行された結果であり、どの配送トランザクションデータを対象としたかを、フォルダ名から把握できるようにする。

### 11.2 保存対象

本 PoC では、以下を保存対象とする。

- OSMデータ
- 前処理生成データ
- 距離・時間計算データ
- 候補案出力データ
- 可視化成果物
- 実行時条件情報
- エラー実行時の実行結果情報

前処理生成データには、少なくとも以下を含む。

- `geocode_result`
- `network_assignment_result`

距離・時間計算データには、以下を含む。

- `od_matrix_result`

候補案出力データには、少なくとも以下を含む。

- `candidate_plan_summary`
- `candidate_route_detail`

可視化成果物には、少なくとも以下を含む。

- 全体ルート図
- ルート別可視化
- 訪問順序付き可視化

### 11.3 OSMデータの保存方針

OSMデータは、道路ネットワークの正本データとして保存する。  
再現性確保の観点から、利用した OSM データは版を識別できる形で保持し、最新版のみを保持するのではなく、旧版も退避保存する。

保存構成は、`osm/` 配下に最新版ファイルを配置し、更新によって不要となった旧版は `old/` フォルダへ移動する方式とする。

構成イメージは以下のとおりとする。

```text
osm/
  kanagawa-latest.osm.pbf
  old/
    kanagawa-20260301.osm.pbf
    kanagawa-20260201.osm.pbf
```

この構成により、日常的な参照は最新版ファイルを利用しつつ、必要に応じて旧版を参照できる状態を保つ。  
また、各実行結果には、利用した `osm_version` を実行時条件情報として記録する。

### 11.4 実行単位フォルダの保存構成

前処理生成データ、距離・時間計算データ、候補案出力データ、可視化成果物、実行時条件情報は、`plan_id` 単位の実行フォルダに保存する。  
実行フォルダ配下では、用途ごとに系統を分けて保存する。

基本構成は以下のとおりとする。

```text
{plan_id}/
  execution_metadata.json
  preprocess/
    geocode_result.json
    network_assignment_result.json
  distance/
    od_matrix_result.json
  output/
    table/
      candidate_plan_summary.json
      candidate_route_detail.json
    image/
      A001/
        overall_route_map.png
        route_map.png
        stop_order_route_map.png
      A002/
        overall_route_map.png
        route_map.png
        stop_order_route_map.png
```

この構成により、1 回の実行結果を一式で確認できるようにする。  
また、候補案ごとの画像成果物は `candidate_plan_id` 単位でフォルダ分割することで、候補案ごとのまとまりを保ちながら視認性を高める。

### 11.5 前処理生成データおよび出力データの保存形式

前処理生成データおよび出力データの保存形式は、JSONを基本とする。  
JSON形式を採用することで、再利用時に構造を保持しやすく、後続処理や簡易的な再読込にも対応しやすくする。

対象は少なくとも以下とする。

- `geocode_result.json`
- `network_assignment_result.json`
- `od_matrix_result.json`
- `candidate_plan_summary.json`
- `candidate_route_detail.json`

可視化成果物については、画像ファイルとして保存する。

### 11.6 実行時条件情報の保存方針

実行結果の再現性と比較可能性を担保するため、各 `plan_id` に対して実行時条件情報を保存する。  
実行時条件情報は、実行結果そのものとは別に、当該実行で参照した前提条件および評価条件を記録するメタデータとして扱う。
これらの実行時条件情報は、`execution_metadata.json` に保存する。

保存項目は少なくとも以下を含む。

- `plan_id`
- `delivery_date`
- `executed_at`
- `osm_version`
- `graph_generated_at`
- `assignment_rule`
- `travel_time_adjustment_condition`
- `vehicle_count_min`
- `vehicle_count_max`
- `fixed_cost_per_vehicle`
- `distance_unit_cost`
- `work_start_time`
- `work_end_time`
- `time_slot_definition`
- `result_status`
- `provisional_reason`
- `geocode_failed_count`
- `network_assignment_failed_count`
- `od_failed_pair_count`
- `status`
- `error_message`

ここで `status` は `success / failed` を保持し、`error_message` は失敗時の概要を保持する。  
これにより、正常実行と異常実行を同一の保存構成で管理しつつ、後から実行状態を確認できるようにする。

### 11.7 エラー実行時の保存方針

本 PoC では、エラーが発生した実行結果についても保存対象とする。  
理由は、エラー発生時の入力条件、前処理結果、実行条件を保持しておかないと、原因調査や再現確認が困難になるためである。

エラー実行時も、正常実行時と同じく `plan_id` 単位でフォルダを作成し、生成済みの前処理データ、中間データ、実行時条件情報を可能な範囲で保存する。  
そのうえで、`execution_metadata.json` に `status = failed` および `error_message` を記録する。

### 11.8 保存期間

本 PoC では、保存データは人為的に削除しない限り永続保持を基本とする。  
PoC段階では、長期的な容量最適化よりも、再現性、比較可能性、検証可能性を優先するためである。

ただし、将来的に本番運用を見据える場合には、容量制約や運用負荷を踏まえ、保存期間やアーカイブ方針を別途定義する余地を残す。
