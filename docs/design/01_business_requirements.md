# 業務要件

このドキュメントでは、VRP PoC における業務上の要件を定義する。

## 1. ドキュメントの目的

本ドキュメントの目的は、`00_problem_definition.md` で定義した VRP PoC の問題設定を、業務で利用価値のある PoC 要件へ落とし込むことである。

本ドキュメントでは、「どのような業務課題に対して」「どのような価値を返す PoC を作るのか」を明確にし、その価値提供に必要となる入力データ、出力データ、最適化の前提条件、評価観点を整理する。

特に本 PoC では、単に最適化モデルを実装すること自体を目的とするのではなく、配送計画担当者、実配送運転手、導入判断者といった利用者が、実道路ネットワークを前提とした配送計画の価値を判断できることを重視する。

そのため、本ドキュメントでは以下を整理対象とする。

- 想定する業務シナリオ
- 解決したい業務課題
- PoC が提供する業務価値
- 業務上意味を持つ入力データと出力データ
- PoC として成立させるために必要な基盤要件
- 業務上の成功基準および PoC の可否判断基準

これにより、本 PoC を「実道路データを使った最適化の技術検証」で終わらせず、業務適用可能性を判断できる社内 PoC レベルの成果物として位置付ける。

## 2. 想定業務シナリオ

本 PoC は、配送計画を前日に作成し、翌日の配送業務へ反映する業務シナリオを想定する。

本 PoC の開始条件は、翌日の配送対象データが確定しており、単一デポに紐づく稼働可能台数レンジといった当日の運用条件を参照できる状態であることとする。

想定する通常時の業務の流れは以下のとおりである。

1. 前日に、翌日の配送対象データが一括で確定する  
   入力データには、配送先住所と各配送先に対応する荷物個数が含まれる。

2. 配送計画担当者が、配送対象データと当日の運用条件を確認し、VRP PoC を実行する  
   稼働可能台数の下限値と上限値は、単一デポに紐づくマスタ情報として保持し、必要に応じてその日の事情に応じた値へ補正できるものとする。

3. VRP PoC が、住所情報の前処理を行ったうえで最適化を実行し、配送ルート案、必要台数、概算コストなどを出力する  
   住所はそのままでは最適化に利用できないため、PoC 側で住所正規化、ジオコーディング、道路ネットワークへの割当を行い、配送計画に利用可能な地点情報へ変換する。このとき使用台数は完全自由に決めるのではなく、当日のシフトや稼働可能人数を踏まえた台数レンジの中で最適化する。

4. 配送計画担当者が、出力結果を確認し、現場都合に応じて台数選択やルート構成、配送順序の調整を行う  
   必要に応じて、台数レンジ内での使用台数の選択、各ルートにどの配送先を持たせるかといったルート構成の調整、配送順序の調整を行い、最終計画へ反映する。

5. 確定した配送計画を翌日の配送担当者へ共有し、実配送を行う  
   ここで PoC の出力は、厳密なナビゲーションそのものではなく、配送順序、移動距離、巡回のまとまりを把握するための業務用計画案として利用する。

例外時は、エラー内容を確認したうえで、必要な修正を行って再実行する。

- 住所不備やジオコーディング失敗の場合は、配送計画担当者が該当住所を修正したうえで再実行する
- 台数レンジや容量制約により実行可能解が得られない場合は、当日の運用条件を見直したうえで再実行する
- 出力ルートが現場感覚と大きく乖離する場合は、代替案の確認または手動調整を行う

このシナリオにおいて、本 PoC が担う役割は、配送先一覧、荷物個数、単一デポに紐づく稼働可能台数レンジをもとに、実道路を前提とした配送計画案を前日に自動生成し、必要台数とルート構成の初期案を提示することである。

また、本 PoC は最終判断を完全自動化するものではなく、計画担当者が経験則や当日の事情を踏まえて最終調整を行う前提の意思決定支援ツールとして位置付ける。

## 3. 解決したい業務課題

本 PoC では、最終的な目的を配送コスト削減に置きつつ、その実現を阻害している業務課題を整理する。ここでは、単に「コストを下げたい」と捉えるのではなく、現場運用と管理判断の両面から、どのような課題が存在し、それに対して本 PoC がどのような価値を返すかを明確にする。

### 3.1 実道路に即した配送計画を作りにくい

- 対象者  
  配送計画担当者、実配送担当者、導入判断者

- 業務課題  
  直線距離や簡易距離を前提とした配送計画では、実際の道路事情を十分に反映できず、必要台数や総距離、コスト見積もりの精度が下がりやすい。

- 具体例  
  一方通行や道路接続の影響により、机上では近く見える配送先でも、実運用上は大きく迂回が必要になる。  
  簡易距離でルートを組むと、現場感覚では不自然な巡回順序になりやすい。

- 本 PoC による解決の方向性  
  OSM の実道路ネットワークを用いて距離とルートを算出することで、机上距離ではなく実道路に即した配送計画案を作成し、必要台数や概算コストの精度向上を目指す。

### 3.2 配送計画作成に時間と人手がかかる

- 対象者  
  配送計画担当者

- 業務課題  
  翌日の配送先件数が多い場合、担当者が人手で配送順序や台数を検討する必要があり、計画作成に時間がかかる。

- 具体例  
  配送先が多い日に、複数ルートの組み方や必要台数の見積もりに時間を要する。  
  台数を変えた場合の比較案を確認したいときに、同様の検討を繰り返す必要がある。

- 本 PoC による解決の方向性  
  配送先住所と荷物個数から、実道路ベースの初期案を自動生成することで、計画作成にかかる初動工数を下げ、担当者が確認と調整に集中できる状態を目指す。

### 3.3 配送計画が属人化しやすく再現性が低い

- 対象者  
  配送計画担当者、導入判断者

- 業務課題  
  配送計画が特定担当者の経験や土地勘に依存すると、担当者が変わったときに計画品質がぶれやすく、再現性が低くなる。

- 具体例  
  同じ配送条件でも、担当者ごとに配送順序や必要台数の判断が変わる。  
  計画根拠が暗黙知化しており、引き継ぎやレビューがしにくい。

- 本 PoC による解決の方向性  
  実道路データと明示的な制約条件に基づいて配送計画案を生成することで、計画作成の土台を標準化し、担当者依存を軽減する。

### 3.4 計画根拠やコスト妥当性を説明しにくい

- 対象者  
  配送計画担当者、導入判断者

- 業務課題  
  なぜその台数なのか、なぜそのルートなのか、なぜそのコスト見積もりなのかを定量的に説明しづらいと、社内での納得形成や導入判断が難しくなる。

- 具体例  
  必要台数の判断が経験則中心で、固定費や距離との関係を説明しにくい。  
  ルート案は提示できても、なぜその巡回順序になったかを管理者や現場へ説明しづらい。

- 本 PoC による解決の方向性  
  使用台数、総走行距離、概算コスト、ルート可視化をセットで出力することで、配送計画の根拠を定量的かつ視覚的に説明できる状態を目指す。

### 3.5 複数案の比較や改善余地を把握しにくい

- 対象者  
  配送計画担当者、導入判断者

- 業務課題  
  現状のやり方では、単一の配送案は作成できても、他の案と比較してどの程度良いのか、改善余地がどこにあるのかを把握しにくい。

- 具体例  
  台数を 1 台増減した場合に、総距離やコストがどの程度変化するかをすぐに比較できない。  
  現状運用に対して、どれだけ台数削減や距離短縮の余地があるのかが見えにくい。

- 本 PoC による解決の方向性  
  単一の最適解だけでなく、近い代替案や比較可能な指標を出力することで、配送計画担当者と導入判断者が改善余地を具体的に把握できる状態を目指す。
