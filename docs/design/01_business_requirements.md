# 業務要件

このドキュメントでは、VRP PoC における業務上の要件を定義する。

## 1. ドキュメントの目的

本ドキュメントの目的は、`00_problem_definition.md` で定義した VRP PoC の問題設定を、業務で利用価値のある PoC 要件へ落とし込むことである。

本ドキュメントでは、「どのような業務課題に対して」「どのような価値を返す PoC を作るのか」を明確にし、その価値提供に必要となる入力データ、出力データ、最適化の前提条件、評価観点を整理する。

特に本 PoC では、単に最適化モデルを実装すること自体を目的とするのではなく、配送計画担当者、実配送運転手、導入判断者といった利用者が、実道路ネットワークを前提とした配送計画の価値を判断できることを重視する。

そのため、本ドキュメントでは以下を整理対象とする。

- 想定する業務シナリオ
- 解決したい業務課題
- PoC が提供する業務価値
- 業務上意味を持つ入力データと出力データ
- PoC として成立させるために必要な基盤要件
- 業務上の成功基準および PoC の可否判断基準

これにより、本 PoC を「実道路データを使った最適化の技術検証」で終わらせず、業務適用可能性を判断できる社内 PoC レベルの成果物として位置付ける。

## 2. 想定業務シナリオ

本 PoC は、配送計画を前日に作成し、翌日の配送業務へ反映する業務シナリオを想定する。

本 PoC の開始条件は、翌日の配送対象データが確定しており、単一デポに紐づく稼働可能台数レンジといった当日の運用条件を参照できる状態であることとする。

想定する通常時の業務の流れは以下のとおりである。

1. 前日に、翌日の配送対象データが一括で確定する。入力データには、配送先住所と各配送先に対応する荷物個数が含まれる。
2. 配送計画担当者が、配送対象データと当日の運用条件を確認し、VRP PoC を実行する。稼働可能台数の下限値と上限値は、単一デポに紐づくマスタ情報として保持し、必要に応じてその日の事情に応じた値へ補正できるものとする。
3. VRP PoC が、住所情報の前処理を行ったうえで最適化を実行し、配送ルート案、必要台数、概算コストなどを出力する。住所はそのままでは最適化に利用できないため、PoC 側で住所正規化、ジオコーディング、道路ネットワークへの割当を行い、配送計画に利用可能な地点情報へ変換する。このとき使用台数は完全自由に決めるのではなく、当日のシフトや稼働可能人数を踏まえた台数レンジの中で最適化する。
4. 配送計画担当者が、出力結果を確認し、現場都合に応じて台数選択やルート構成、配送順序の調整を行う。必要に応じて、台数レンジ内での使用台数の選択、各ルートにどの配送先を持たせるかといったルート構成の調整、配送順序の調整を行い、最終計画へ反映する。
5. 確定した配送計画を翌日の配送担当者へ共有し、実配送を行う。ここで PoC の出力は、厳密なナビゲーションそのものではなく、配送順序、移動距離、巡回のまとまりを把握するための業務用計画案として利用する。

例外時は、エラー内容を確認したうえで、必要な修正を行って再実行する。

- 住所不備やジオコーディング失敗の場合は、配送計画担当者が該当住所を修正したうえで再実行する
- 台数レンジや容量制約により実行可能解が得られない場合は、当日の運用条件を見直したうえで再実行する
- 出力ルートが現場感覚と大きく乖離する場合は、代替案の確認または手動調整を行う

このシナリオにおいて、本 PoC が担う役割は、配送先一覧、荷物個数、単一デポに紐づく稼働可能台数レンジをもとに、実道路を前提とした配送計画案を前日に自動生成し、必要台数とルート構成の初期案を提示することである。

また、本 PoC は最終判断を完全自動化するものではなく、計画担当者が経験則や当日の事情を踏まえて最終調整を行う前提の意思決定支援ツールとして位置付ける。

## 3. 解決したい業務課題

本 PoC では、最終的な目的を配送コスト削減に置きつつ、その実現を阻害している業務課題を整理する。ここでは、単に「コストを下げたい」と捉えるのではなく、現場運用と管理判断の両面から、どのような課題が存在し、それに対して本 PoC がどのような価値を返すかを明確にする。

### 3.1 実道路に即した配送計画を作りにくい

対象者：配送計画担当者、実配送担当者、導入判断者  
業務課題：直線距離や簡易距離を前提とした配送計画では、実際の道路事情を十分に反映できず、必要台数や総距離、コスト見積もりの精度が下がりやすい。  
具体例：一方通行や道路接続の影響により、机上では近く見える配送先でも、実運用上は大きく迂回が必要になる。簡易距離でルートを組むと、現場感覚では不自然な巡回順序になりやすい。  
本 PoC による解決の方向性：OSM の実道路ネットワークを用いて距離とルートを算出することで、机上距離ではなく実道路に即した配送計画案を作成し、必要台数や概算コストの精度向上を目指す。

### 3.2 配送計画作成に時間と人手がかかる

対象者：配送計画担当者  
業務課題：翌日の配送先件数が多い場合、担当者が人手で配送順序や台数を検討する必要があり、計画作成に時間がかかる。  
具体例：配送先が多い日に、複数ルートの組み方や必要台数の見積もりに時間を要する。台数を変えた場合の比較案を確認したいときに、同様の検討を繰り返す必要がある。  
本 PoC による解決の方向性：配送先住所と荷物個数から、実道路ベースの初期案を自動生成することで、計画作成にかかる初動工数を下げ、担当者が確認と調整に集中できる状態を目指す。

### 3.3 配送計画が属人化しやすく再現性が低い

対象者：配送計画担当者、導入判断者  
業務課題：配送計画が特定担当者の経験や土地勘に依存すると、担当者が変わったときに計画品質がぶれやすく、再現性が低くなる。  
具体例：同じ配送条件でも、担当者ごとに配送順序や必要台数の判断が変わる。計画根拠が暗黙知化しており、引き継ぎやレビューがしにくい。  
本 PoC による解決の方向性：実道路データと明示的な制約条件に基づいて配送計画案を生成することで、計画作成の土台を標準化し、担当者依存を軽減する。

### 3.4 計画根拠やコスト妥当性を説明しにくい

対象者：配送計画担当者、導入判断者  
業務課題：なぜその台数なのか、なぜそのルートなのか、なぜそのコスト見積もりなのかを定量的に説明しづらいと、社内での納得形成や導入判断が難しくなる。  
具体例：必要台数の判断が経験則中心で、固定費や距離との関係を説明しにくい。ルート案は提示できても、なぜその巡回順序になったかを管理者や現場へ説明しづらい。  
本 PoC による解決の方向性：使用台数、総走行距離、概算コスト、ルート可視化をセットで出力することで、配送計画の根拠を定量的かつ視覚的に説明できる状態を目指す。

### 3.5 複数案の比較や改善余地を把握しにくい

対象者：配送計画担当者、導入判断者  
業務課題：現状のやり方では、単一の配送案は作成できても、他の案と比較してどの程度良いのか、改善余地がどこにあるのかを把握しにくい。  
具体例：台数を 1 台増減した場合に、総距離やコストがどの程度変化するかをすぐに比較できない。現状運用に対して、どれだけ台数削減や距離短縮の余地があるのかが見えにくい。  
本 PoC による解決の方向性：単一の最適解だけでなく、近い代替案や比較可能な指標を出力することで、配送計画担当者と導入判断者が改善余地を具体的に把握できる状態を目指す。

## 4. PoC が提供する業務価値

本 PoC では、前章で整理した業務課題に対して、実道路ネットワークを前提とした配送計画案を返すことで、現場運用と管理判断の両方に価値を提供する。ここでは、各業務課題に対して本 PoC がどのような価値を返すかを明確にする。

### 4.1 実道路に即した配送計画案を得られる価値

対象者：配送計画担当者、実配送担当者、導入判断者  
提供価値：実道路ネットワークに基づく配送ルート案と距離情報を返すことで、机上距離ではなく現実の道路事情に近い配送計画案を作成できる。  
具体的に得られるもの：実道路ベースの配送順序、各区間距離および各ルート総距離、ルート可視化による巡回イメージ。  
この価値が有効である理由：直線距離や簡易距離では把握しづらい一方通行や道路接続の影響を反映できるため、必要台数やルート妥当性の判断精度を高めやすい。

### 4.2 配送計画作成の初動工数を下げられる価値

対象者：配送計画担当者  
提供価値：配送先住所、荷物個数、台数条件を入力として、配送計画の初期案を自動生成することで、担当者が人手でゼロから計画を組む負担を減らせる。  
具体的に得られるもの：必要台数の初期案、ルートごとの配送順序、台数条件を踏まえたルート構成案。  
この価値が有効である理由：担当者は初期案の作成そのものではなく、現場都合を踏まえた確認と調整に時間を使えるようになるため、前日計画業務の効率化につながる。

### 4.3 配送計画の標準化と再現性を高められる価値

対象者：配送計画担当者、導入判断者  
提供価値：経験や土地勘だけに依存せず、同じ入力条件に対して一定のルールで配送計画案を生成することで、計画作成の標準化と再現性向上に寄与する。  
具体的に得られるもの：明示的な入力条件に基づく計画案、担当者が変わっても再利用できる計画作成の土台、調整前後を比較しやすい初期案。  
この価値が有効である理由：計画品質が特定担当者に依存しにくくなり、引き継ぎやレビューがしやすくなるため、業務運用の安定化に寄与する。

### 4.4 計画根拠とコスト妥当性を説明できる価値

対象者：配送計画担当者、導入判断者  
提供価値：使用台数、総走行距離、概算コスト、ルート可視化をセットで返すことで、配送計画の根拠を定量的かつ視覚的に説明できる。  
具体的に得られるもの：台数と距離に基づく概算コスト、ルート案ごとの比較可能な指標、配送順序と巡回構造を示す可視化結果。  
この価値が有効である理由：「なぜその台数なのか」「なぜその順序なのか」を説明しやすくなり、社内の納得形成や導入可否判断を進めやすくなる。

### 4.5 複数案を比較し、改善余地を把握できる価値

対象者：配送計画担当者、導入判断者  
提供価値：単一の最適解だけでなく、近い代替案や比較可能な指標を返すことで、配送計画の改善余地を把握しやすくする。  
具体的に得られるもの：台数やルート構成が異なる複数案、各案の総距離、概算コスト、必要台数、現場調整の判断材料。  
この価値が有効である理由：最適化結果をそのまま採用するだけでなく、運用都合や現場判断を踏まえた比較検討が可能になるため、実務導入しやすい計画支援ツールとして利用しやすくなる。

## 5. 提供する VRP モデルの業務要件

本章では、前章までに整理した業務課題と提供価値を、実際にどのような VRP モデルとして業務へ適用するかという観点で整理する。ここで扱うのは数式レベルの詳細設計ではなく、業務上必要な配送計画の条件を、最適化モデルとしてどう表現するかの前提である。

### 5.1 前提配送計画

本 PoC では、前日に翌日分の配送計画を作成する運用を前提とする。  
入力となる配送対象データは前日時点で確定しており、その情報をもとに翌日の配送計画案を作成する。

配送計画は単一デポ単位で扱い、各車両はデポを起点として出発し、配送完了後は同じデポへ帰着する。  
また、当日配送対象となった荷物は、その日のうちに全量配送することを前提とする。

### 5.2 業務制約条件

本 PoC では、業務上守るべき条件として以下を前提とする。

- 当日配送対象となった配送先は全件訪問する
- 各配送先に割り当てられた荷物数は全量配送する
- 各車両の積載量は車両容量の上限を超えない
- 使用台数は、その日に稼働可能な台数レンジの範囲内とする
- 各車両はデポに帰着した時点で業務終了とする
- 各車両は 1 日 8 時間以内にデポへ戻ることを前提とする
- 各荷物が持つ `午前 / 午後 / 時間指定なし` の時間帯区分に応じて、その時間帯に配送できるようにする

### 5.3 最適化対象

本 PoC では、業務上の主目的をコスト削減に置く。  
コストは簡略化して、`使用台数 × 1台あたり固定コスト + 総走行距離 × 距離比例コスト` として表現する。

このうち、業務上は 1 台あたり固定コストの影響が大きいと想定するため、結果として使用台数を抑えることを強く重視する。  
そのうえで、同じ台数条件の中では総走行距離を抑え、距離比例コストを下げることを第二の最適化対象とする。

### 5.4 モデル前提

本 PoC で扱う VRP モデルは、業務要件を簡略化して以下の前提で表現する。

- 需要量は荷物個数で表現する
- 車両容量も荷物個数で表現する
- 距離は OSM の実道路ネットワークを用いて算出する
- 使用台数は固定値ではなく、入力された台数レンジの中で決定する
- 配送先住所は、そのままでは最適化に利用せず、PoC 側で住所正規化、ジオコーディング、道路ネットワークへの割当を行ったうえで利用する
- 入力として与えられる配送情報は、住所情報、荷物個数、配送日を基本とする
- 台数コストは単一デポに紐づくパラメータとして扱う

### 5.5 入出力の位置づけ

本 PoC における入力と出力の位置づけは以下のとおりとする。

入力として必要なもの

- 配送先情報
- 各配送先に紐づく荷物個数
- 配送日
- 単一デポ情報
- 稼働可能台数レンジ
- デポに紐づく台数コスト
- 距離比例コスト

出力として返すもの

- 必要台数
- 配送順序を含むルート案
- 概算コスト
- 地図上で確認できるルート可視化
- 台数レンジ内で比較可能な代替案
- 各ルートの総距離
- 各区間距離
- `午前 / 午後 / 時間指定なし` のどの時間帯で配送する想定かを示す到着時間帯情報

ルート案の出力では、配送担当者が巡回順序を理解できるよう、各配送先に訪問順を付番した状態で確認できることを前提とする。  
また、一覧形式の出力では、`配送先1`、`配送先2` といった形で訪問順序を確認できる構成を想定する。

代替案は、台数レンジの中で異なる台数案やルート構成案を比較できる形で出力し、配送計画担当者が当日の運用条件や現場都合に応じて採用案を判断できるようにする。
