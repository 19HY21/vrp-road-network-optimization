# 実験設計

このドキュメントでは、`05_algorithm_design.md` で定義したアルゴリズム設計を実装する上で行う検証の設計を定義する。

---

## 1. ドキュメントの目的

本ドキュメントは、`00_problem_definition.md` から `05_algorithm_design.md` にかけて設計した内容が正しく機能することを確認するための検証設計を記録することを目的とする。

具体的には、以下の 4 点を定義する。

- **ソルバー選定の検証設計**：`05_algorithm_design.md` の 2.4 節で検証が必要とされているソルバー採用判断を行うための実験条件・手順・判定基準を定義する
- **計算時間上限の検証設計**：同 5.4 節で検証が必要とされている探索時間上限の確定に向けた実験条件・手順・判定基準を定義する
- **解の妥当性検証設計**：実装によって得られた最適化解が現実の道路ネットワーク・配送要件に対して妥当かを確認するための検証方法を定義する
- **異常状態の出力形式**：同 7.4 節で形式の定義が委ねられている、異常状態発生時の記録フォーマットを定義する

本ドキュメントは実験の結果や実装コードを含まない。検証をどのように設計するかを定義することを目的とし、各検証の実施結果は `05_algorithm_design.md` の対応する節に記載する。

---

## 2. ソルバー選定の検証設計

本章では、`05_algorithm_design.md` の 2.4 節で定義したソルバー選定を確定するための実験設計を定義する。同節では「何を確認するか・採用基準の構造」を定義しており、本章では「どうやって確認するか・どう記録するか」の具体を定める。

### 2.1 テストケースの定義

| 項目 | 内容 |
|---|---|
| 対象地点数 | 30〜50 地点 |
| 実座標の取得方法 | 神奈川県内の実在する住所を対象に、OSMnx を用いて最近傍ノードの座標を取得する |
| 適用する制約水準 | `04_math_model.md` の第 7 章で定義した 11 の制約条件すべてを適用する |
| デポ設定 | 本 PoC で想定する単一デポを使用する |

テストケースに用いる地点は、候補ライブラリ間の比較を公平にするために全ライブラリで共通とする。

### 2.2 実験条件と実行手順

#### 統一探索時間

全候補ライブラリに対して探索時間上限を **30 秒** に統一する。この値は第 3 章で定義する計算時間検証の第 1 段階（30 秒）と同一条件であり、章間での実験条件の一貫性を確保するために採用する。

#### 実行手順

各候補ライブラリを `05_algorithm_design.md` の 2.3 節で定義した優先順位（OR-Tools Routing Library → OR-Tools CP-SAT → Python-MIP → PuLP → Pyomo）の順に実行する。

1. テストケース（地点座標・OD 行列・制約パラメータ）を共通フォーマットで用意する
2. 対象ライブラリに探索時間上限 30 秒を設定した上でテストケースを投入する
3. 実行終了後、ソルバーが返すステータスと解を記録する
4. 必須条件（後述 2.4 節）を満たした時点でそのライブラリを採用候補として確定し、以降の候補の検証は省略できる

#### 再現性の確保

乱数シードを固定した上で実行し、同一条件での再実行が同じ結果を返すことを確認する。固定値は実装時に決定する。

### 2.3 計測指標と記録形式

実行結果を以下の指標で計測し、ライブラリごとに記録する。

| 指標 | 計測方法 |
|---|---|
| 解の有無 | ソルバーが返すステータスコードを参照して自動判定する |
| 制約充足 | 得られた解に対して 11 制約の充足を自動検証する |
| 目的関数値（総コスト） | 解から `車両固定費 × 使用台数 + 距離単価 × 総走行距離` を計算して記録する |
| 実行時間 | 探索開始から終了までの実壁時間（秒）を計測する |
| 目視確認 | 得られたルートを地図上にプロットし、経路の現実的妥当性を人間が確認する |

記録は以下の表形式で各ライブラリの結果を横断比較できるよう整理する。

| ライブラリ | 解の有無 | 制約充足 | 目的関数値 | 実行時間（秒） | 目視確認 |
|---|---|---|---|---|---|
| OR-Tools Routing Library | - | - | - | - | - |
| OR-Tools CP-SAT | - | - | - | - | - |
| Python-MIP | - | - | - | - | - |
| PuLP | - | - | - | - | - |
| Pyomo | - | - | - | - | - |

### 2.4 採用判定の基準と結果記録

#### 採用判定の基準

以下の優先順位で採用ライブラリを決定する。

1. **必須条件**：解が返ること、かつ 11 制約すべてを満たしていること。いずれか一方でも満たさない場合は失格とする
2. **解の品質**：必須条件を満たした候補が複数ある場合は、統一探索時間内の目的関数値が小さい方を上位とする
3. **現実性確認**：目視確認により経路が現実的であることを確認する

#### 結果記録のフォーマット

検証完了後、以下の問いに答える形式で採用結果を `05_algorithm_design.md` の 2.4 節に記載する。

| 記録項目 | 内容 |
|---|---|
| 採用ライブラリ | 採用したライブラリ名 |
| 採用理由 | 解の有無・制約充足・目的関数値・目視確認の結果をもとに記載する |
| 不採用理由 | 各候補を不採用とした理由を記載する |
| 目視確認の結果 | 経路の現実的妥当性について確認した内容を記載する |
| 全制約実装の根拠 | 採用ライブラリで 11 制約を実装できると判断した根拠を記載する |

## 3. 計算時間上限の検証設計

本章では、`05_algorithm_design.md` の 5.4 節で定義した計算時間の 3 段階検証を実施するための実験設計を定義する。目的は「同じ問題に対して探索時間を変えたとき、どの時点で解の改善が頭打ちになるか」を実証することであり、採用する探索時間上限を確定する根拠とする。

### 3.1 実験条件

| 項目 | 内容 |
|---|---|
| 前提 | 第 2 章のソルバー選定が完了し、採用ソルバーが確定した後に実施する |
| 使用するテストケース | 第 2 章で使用したものと同一のテストケースを使用する |
| 使用するソルバー | 第 2 章で採用が確定したソルバー |
| 探索時間上限 | 30 秒・120 秒・300 秒の 3 段階（`05_algorithm_design.md` 5.4 節に対応） |
| 実行回数 | 各段階につき同一条件で 1 回実行する |
| 再現性の確保 | 第 2 章と同一の乱数シードを使用する |

同一テストケースを使用する理由は、探索時間の違いによる解の変化のみを比較対象とするためである。テストケースが異なると時間効果と問題の違いが混在し、頭打ち点の判断が成立しない。

### 3.2 計測方法と記録形式

各段階で以下の指標を計測する。

| 指標 | 計測方法 |
|---|---|
| 目的関数値（総コスト） | 各段階終了時点の最良解から `車両固定費 × 使用台数 + 距離単価 × 総走行距離` を計算する |
| 実行時間 | 探索開始から終了までの実壁時間（秒）を計測する |

記録は以下の表形式で各段階の結果と段階間の改善幅を整理する。

| 段階 | 探索時間上限（秒） | 目的関数値 | 実行時間（秒） | 前段階からの改善幅 |
|---|---|---|---|---|
| 短 | 30 | - | - | - |
| 中 | 120 | - | - | - |
| 長 | 300 | - | - | - |

改善幅は「前段階の目的関数値 − 当該段階の目的関数値」として算出する。値が小さいほど改善が頭打ちに近いことを示す。

### 3.3 採用時間上限の判定基準と結果記録

#### 判定基準

以下の基準で採用する探索時間上限を決定する。

- 段階間の改善幅が前段階と比較して顕著に小さくなった場合、その手前の段階を採用候補とする
- 全段階を通じて改善幅が継続して大きい場合は、最長の 300 秒を採用候補とする
- 最短の 30 秒で十分な解が得られており以降の改善が軽微な場合は、30 秒を採用候補とする

#### 結果記録のフォーマット

検証完了後、以下の問いに答える形式で結果を `05_algorithm_design.md` の 5.5 節に記載する。

| 記録項目 | 内容 |
|---|---|
| 各段階の目的関数値 | 30 秒・120 秒・300 秒それぞれの目的関数値を記載する |
| 段階間の改善の傾向 | 計算時間を延ばすにつれて解の改善が確認できたかを記載する |
| 頭打ちが確認された段階 | 改善幅が顕著に小さくなった段階を記載する |
| 採用する探索時間上限 | 採用した時間上限とその判断根拠を記載する |

## 4. 解の妥当性検証設計

本章では、実装によって得られた最適化解が現実の道路ネットワーク・配送要件に対して妥当かを確認するための検証設計を定義する。「何を確認するか」と「どのような手法で確認するか」をセットで定める。

### 4.1 ルートの目視確認

#### 確認する対象

VRP ソルバーが返したルートの経路が、現実の道路に沿って物理的に成立しているかを確認する。具体的には、迂回路のない自然な経路になっているか、明らかに非現実的な経路が含まれていないかを判断する。

#### 確認手法

得られたルートの訪問順に従い、各地点の座標を地図ツール（Google Maps・Apple Maps 等）上に順序付きでプロットする。経路を視覚的に確認し、道路の連続性と経路の現実的妥当性を人間が判断する。

#### 確認基準

以下のいずれかに該当する場合を問題ありと判定する。

- 道路が存在しない区間を経路として含んでいる
- 同一地点への重複訪問や明らかな遠回りが生じている
- デポへの出発・帰着が正しく反映されていない

### 4.2 外部参照との距離比較

#### 確認する対象

OSMnx を用いて計算した区間ごとの道のり距離が、外部参照と比較して著しい乖離がないかを確認する。距離値の精度は目的関数値（総コスト）の信頼性に直結するため、乖離の有無を実証的に確認する。

#### 確認手法

OD 行列から代表的な区間を抽出し、Google Maps 等の地図サービスで同区間の距離を手動で照合する。全区間の網羅的な比較は行わず、距離が長い区間・短い区間・デポを起終点とする区間を各数件抽出して比較対象とする。

#### 確認基準

抽出した区間において、OSMnx の計算値と外部参照との乖離が概ね 10% 以内であることを妥当の目安とする。著しい乖離が見られた場合は、OSMnx のネットワーク取得設定またはノードマッピングに問題がないかを確認する。

### 4.3 制約充足の確認

#### 確認する対象

出力された解に対して、`04_math_model.md` の第 7 章で定義した 11 の制約条件がすべて充足されているかを確認する。

#### 確認手法

解の充足確認はプログラムによる自動検証を主手段とする。各制約について、解から得られる数値（需要合計・到着時刻・訪問回数等）を計算し、制約条件の上下限と照合するスクリプトを実装して判定する。

自動検証が困難な確認（時間帯制約の現実的妥当性など）については、4.1 の目視確認と組み合わせて補完的に判断する。

#### 確認基準

11 制約すべての充足が自動検証により確認できることを合格条件とする。いずれかの制約に違反が検出された場合は、違反した制約と該当ルートを記録し、実装の修正対象として扱う。

### 4.4 記録形式と問題発生時の対処方針

#### 記録形式

各確認項目の結果を Excel の 1 枚目のシートに以下のテーブル形式で記録する。

| 確認項目 | 確認手法 | 結果 | 問題の有無 | 備考 |
|---|---|---|---|---|
| ルートの目視確認 | 地図ツールでのプロット確認 | - | - | - |
| 外部参照との距離比較 | Google Maps 等との手動照合 | - | - | - |
| 制約充足の確認 | 自動検証スクリプト | - | - | - |

ルートの目視確認については、地図ツール上でのプロット画像を PNG として取得し、同一 Excel ブック内の以降のシートにエビデンスとして格納する。

#### 問題発生時の対処方針

| 問題の種類 | 対処方針 |
|---|---|
| ルートが物理的に成立しない | OSMnx のネットワーク取得範囲またはノードマッピングの設定を見直す |
| 距離値に著しい乖離がある | OD 行列の計算設定（重みパラメータ等）を確認し、再計算する |
| 制約違反が検出された | 違反した制約の実装を修正し、再検証を行う |

## 5. 異常状態の出力形式

本章では、`05_algorithm_design.md` の 7.4 節で定義が委ねられている異常状態発生時の記録フォーマットを定義する。同節では「発生した状態の種別と要因候補を記録する」とあり、本章でその具体的なフィールド定義と出力形式を定める。

### 5.1 記録単位と必須フィールド

異常状態の記録は**台数設定ごとに 1 レコード**として出力する。`05_algorithm_design.md` の第 3 章で定義した台数ごとの実行において、いずれかの台数設定でスキップが発生した場合に本レコードを生成する。

必須フィールドは以下の通りとする。

| フィールド名 | 型 | 内容 |
|---|---|---|
| `vehicle_count` | 整数 | 異常状態が発生した台数設定値 |
| `status` | 文字列 | 状態種別（`infeasible` / `no_solution_in_time` / `input_error`） |
| `factor_candidates` | 文字列の配列 | 要因として該当する可能性のある候補。特定できない場合は複数列挙する |
| `detected_at` | 文字列 | 検知箇所（`solver_result`：ソルバー実行後 / `pre_validation`：実行前バリデーション） |
| `solver_status_code` | 文字列 | ソルバーが返したステータスコード。入力不備の場合は空文字とする |

### 5.2 出力フォーマット

異常状態の記録は以下の 2 形式で出力する。

#### JSON（機械可読）

プログラムからの出力として生成する。台数設定ごとのレコードを配列として格納する。

```json
[
  {
    "vehicle_count": 3,
    "status": "infeasible",
    "factor_candidates": [
      "稼働可能台数レンジ内の最大台数でも全件訪問が成立しない",
      "時間帯制約と移動時間の組み合わせが成立しない"
    ],
    "detected_at": "solver_result",
    "solver_status_code": "ROUTING_INFEASIBLE"
  }
]
```

#### Excel（人間可読）

確認・共有用として、JSON と同一フィールドを 1 行 1 レコードのテーブル形式でシートに記録する。

| vehicle_count | status | factor_candidates | detected_at | solver_status_code |
|---|---|---|---|---|
| 3 | infeasible | 稼働可能台数レンジ内の最大台数でも全件訪問が成立しない / 時間帯制約と移動時間の組み合わせが成立しない | solver_result | ROUTING_INFEASIBLE |

### 5.3 出力サンプル

`05_algorithm_design.md` の 7.1 節で定義した 3 種類の状態それぞれに対応するサンプルを以下に示す。

#### 実行不可能解（infeasible）

```json
{
  "vehicle_count": 3,
  "status": "infeasible",
  "factor_candidates": [
    "稼働可能台数レンジ内の最大台数でも全件訪問が成立しない",
    "時間帯制約と移動時間の組み合わせが成立しない"
  ],
  "detected_at": "solver_result",
  "solver_status_code": "ROUTING_INFEASIBLE"
}
```

#### 時間内解なし（no_solution_in_time）

```json
{
  "vehicle_count": 4,
  "status": "no_solution_in_time",
  "factor_candidates": [
    "探索時間上限内に実行可能解が得られなかった。探索時間の延長または制約の見直しを検討する"
  ],
  "detected_at": "solver_result",
  "solver_status_code": "ROUTING_FAIL"
}
```

#### 入力不備（input_error）

```json
{
  "vehicle_count": 5,
  "status": "input_error",
  "factor_candidates": [
    "OD 行列に欠損値が含まれている"
  ],
  "detected_at": "pre_validation",
  "solver_status_code": ""
}
```
