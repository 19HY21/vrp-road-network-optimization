# 問題定義

このドキュメントでは、本プロジェクトで扱う VRP PoC の問題設定を定義する。

## 1. 目的

本ポートフォリオの目的は、実道路ネットワークを用いた配送最適化（以下、VRP とする）PoC を通じて、データ前処理、経路距離算出、制約条件整理、最適化モデル設計、評価までを一貫して実装できることを示すことである。

あわせて、BI 領域で培った業務理解やデータ設計の経験を土台に、分析・最適化まで含めた意思決定支援に取り組めることを示す。

## 2. 背景

配送ルート算出では、机上距離や直線距離をもとに概算値を出すケースが見受けられる。展示会で配送ルート計画システムの説明を確認した際にも、実道路に基づく算出を前提としていないケースが見られた。

その場合、実際の道路事情を十分に反映できず、実道路距離との乖離が生じやすい。特に配送のように道路事情や経路制約の影響を大きく受ける業務では、現実の道路ネットワークを踏まえた距離算出やルート探索が重要になる。

そのため、本ポートフォリオでは OSM を用いて道路情報を取得し、実道路に即したルート検索および配送最適化の基盤を構築できるかを検証する。

## 3. 課題

配送業務における利益改善の方向性としては、売上向上とコスト削減が考えられるが、本 PoC ではコスト削減に重点を置く。

配送計画においてコストを下げるためには、少ない台数で効率よく配送ルートを構築し、固定費と走行コストの双方を抑える必要がある。

具体的には、使用車両台数を削減することで、人件費や車両稼働に伴う固定費を抑えられる。また、ルートを最適化して総走行距離を短縮することで、燃料費などの距離比例費も抑えられる。

本 PoC では、こうした配送コスト削減の考え方を、実道路ネットワークに基づく VRP として定式化できるかを課題として扱う。

## 4. 事前検証確認事項

- 神奈川県を対象として OSM による道路ネットワークが取得できることを確認した。Notebook では県全体の `drive` ネットワークを取得し、ノード数 `223,850`、エッジ数 `608,205` の道路グラフを生成できた。  
  参照: `notebooks/001_osm_network_feasibility_validation_plan.ipynb`、`docs/brainstorming/001_osm_network_feasibility_validation_plan.md` の Step 2、`data/raw/road_network/kanagawa_drive.graphml`
- 取得したネットワークで実道路に即した情報が保持されていることを確認した。具体的には `highway`、`oneway`、`length`、`maxspeed`、`access`、`lanes`、`bridge`、`tunnel`、`street_count` など、配送ルート計画や制約検討に使える属性を確認した。  
  参照: `notebooks/001_osm_network_feasibility_validation_plan.ipynb` の `step7-attribute-inventory`、`outputs/validation/step7/kanagawa_attribute_inventory_edges.csv`、`outputs/validation/step7/kanagawa_attribute_inventory_nodes.csv`
- 取得したネットワークをローカルに保存し、再実行時にローカルファイルから呼び出せることを確認した。GraphML 保存後に再読込してノード数・エッジ数が一致することを確認しており、再現可能な入力基盤として扱える。  
  参照: `notebooks/001_osm_network_feasibility_validation_plan.ipynb` の Step 2 および Step 2.5、`docs/brainstorming/001_osm_network_feasibility_validation_plan.md` の Step 2 / Step 2.5、`outputs/metrics/kanagawa_drive_network_metrics.csv`
- 道路ネットワーク上で OD 間の道のり距離を安定して算出できることを確認した。代表地点 `30 OD` と全地点 `132 OD` の比較では、一括取得版と正本グラフ切り出し版で距離・時間・ノード数がすべて一致した。  
  参照: `notebooks/001_osm_network_feasibility_validation_plan.ipynb` の `step5-route-validation`、`step5-all-route-validation`、`outputs/metrics/kanagawa_step5_route_differences.csv`、`outputs/metrics/kanagawa_step5_all_route_differences.csv`
- 自治体ごとに独立取得した道路ネットワークを後から結合する方式も検証したが、県全体 routing 用の基盤としては不適と判断した。理由として、都道府県全体を分割した断片を再結合しているのではなく、自治体ごとに別々の OSM グラフを取得しているため、再結合後に `No path` が発生し、node / edge の整合性も維持できなかった。  
  参照: `notebooks/001_osm_network_feasibility_validation_plan.ipynb` の Step 3 / Step 4、`docs/brainstorming/001_osm_network_feasibility_validation_plan.md` の Step 3 / Step 4
- Google Maps 比較を通して、距離と所要時間の利用可能範囲を整理できることを確認した。距離は順位一致率が高く PoC の目的関数として利用可能と判断し、所要時間は静的推定値としては使えるが、補正なしでは実時間の代替として使うべきではないと判断した。  
  参照: `notebooks/001_osm_network_feasibility_validation_plan.ipynb` の Step 6、`outputs/validation/step6/kanagawa_step6_30od_descriptive_stats.csv`、`outputs/validation/step6/kanagawa_step6_30od_ranking_summary.csv`、`docs/brainstorming/001_osm_network_feasibility_validation_plan.md` の Step 6
- 事前検証結果を踏まえて、VRP PoC の目的関数と制約条件を定義できることを確認した。現時点では「総コスト = 車両固定費 × 台数 + 距離単価 × 総走行距離」を目的関数とし、所要時間は補正前提の制約または参考値として扱う方針を整理した。  
  参照: `docs/brainstorming/001_osm_network_feasibility_validation_plan.md` の Step 7 / Step 8

## 5. 想定利用者

### 5.1 配送ルート計画作成担当者

配送ルート計画を作成する担当者は、限られた車両台数の中で、どの配送先をどの順番で回るかを決める役割を持つ。

本 PoC では、実道路ネットワークに基づく配送順序や総走行距離を確認し、より効率的な配送計画案を作成するために利用することを想定する。

想定ユースケース

- 配送先一覧を入力し、走行距離が短くなる配送ルート案を確認する
- 使用台数を増減させた場合に、総距離やコストがどう変化するかを比較する
- 机上距離ではなく実道路距離に基づく配送順序を検討する

### 5.2 実配送運転手

実配送を担当する運転手は、作成された配送ルートに従って配送を実行する役割を持つ。

本 PoC では、実道路に即した訪問順序や移動距離を確認し、現場感覚と大きく乖離しないルートかを確認するために利用することを想定する。

想定ユースケース

- 配送順序が実際の走行感覚に照らして不自然でないか確認する
- 一方通行や道路構造を踏まえたルートになっているか確認する
- 現場視点で通行しにくい区間や改善余地をフィードバックする

### 5.3 システム導入判断者

システム導入権限を持つ管理者は、PoC の結果を見て、業務適用の可否を判断する役割を持つ。

本 PoC では、実道路データを用いた最適化が、利用に耐える精度・説明可能性・再現性を持つかを確認するために利用することを想定する。

想定ユースケース

- 実道路距離ベースで配送計画を作れるかを確認する
- 使用台数や走行距離の削減余地を確認する
- 今後の本格実装や追加投資に進むかを判断する

## 6. 対象範囲

- 地理範囲：神奈川県を対象とする。理由としては、自身にゆかりのある場所が多く土地勘を持ちやすいことに加え、東京都ほど複雑ではない一方で、都市部・郊外を含み検証対象として十分な規模を持つためである。
- 地理データ：道路ネットワークには、OSM を用いて神奈川県全体から一括取得した道路データを用いる。
- 利用データ制約：道路ネットワークや比較用データは、無償または低コストで取得・再利用できる範囲に限定する。高額な商用 API や常時課金を前提とするデータ基盤は、今回の対象外とする。そのため、本 PoC では道路ネットワークの正本として OSM を採用し、Google Maps などの商用サービスは比較検証用途に限定して扱う。
- 運用前提：ラストマイル配送を想定し、神奈川県内で完結する配送を対象とする。まず 1 県を対象とした理由は、対象範囲を適切に制御しながら、実道路ネットワーク、配送順序、車両台数、道路属性の影響を一体で検証するためである。特に県内配送は、幹線輸送よりも一般道の影響を受けやすく、一方通行、道路種別、幅員、都市部と郊外の差など、実道路制約を踏まえた最適化の検証対象として適している。そのため、本 PoC では県をまたぐ長距離輸送は扱わず、まずは単一都道府県内で成立するかを優先して検証する。
- 配送体制：本 PoC では `人数 = 台数` とし、1 人が 1 台を担当する前提とする。これは、実務上 1 人で 1 台を担当する配送形態が多く見られることに加え、アルバイト経験上、複数名乗車のケースでも全員が配送作業者ではなく、路上駐車対応などを目的とした同乗が含まれる場合があったためである。そのため、PoC では配送作業主体を 1 台あたり 1 人としてモデル化する。
- 最適化対象：コスト削減に重点を置き、使用車両台数の削減を第一の最適化対象とする。あわせて、総走行距離を短縮し、距離比例費を抑えることを第二の最適化対象とする。
- 出力対象：配送ルートの可視化ファイルと、配送計画の判断に必要な指標を出力対象とする。具体的には、運転手が走行経路の当たりを付けられるよう、ある程度拡大した状態で確認できるルート画像を含める。あわせて、配送順序、各区間距離、各ルートの総距離、必要台数、概算コストを出力する。所要時間については、現時点では補正前提の参考値または制約候補として扱い、可能であれば午前 / 午後レベルの到着見込みを出力対象に含める。
- 計画対象期間：前日に VRP を実行し、翌日の配送計画案を出力する運用を想定する。
- 利用レベル：本ポートフォリオでは、社内 PoC レベルの水準を想定した検証と設計を対象とする。すなわち、実務に近い前提を置きつつも、本番運用ではなく、導入可否や実装可能性を判断するための利用レベルにとどめる。

## 7. 対象外

- 神奈川県をまたぐ配送計画や、複数都道府県を対象とした広域配送最適化  
  理由：本 PoC は、まず単一都道府県内で実道路ネットワークを用いた配送最適化が成立するかを確認することを優先しているため。広域配送はネットワーク範囲、時間モデル、運用前提が大きく変わるため、次段階の課題とする。

- 渋滞、事故、時間帯ごとの混雑状況をリアルタイムに反映した動的な所要時間最適化  
  理由：今回利用する OSM ベースの道路データは静的な道路ネットワークを前提としており、リアルタイム交通情報を常時利用する構成は、PoC のコスト前提および再現性の観点から対象外とするため。

- 分単位の到着時刻保証や、厳密な時間指定配送の実現  
  理由：所要時間は現時点では補正前提の参考値または制約候補として扱う段階であり、実交通を前提とした厳密な到着保証まで行える精度は今回の検証範囲に含めないため。

- 車種ごとの詳細制約、荷姿、駐車可否、顧客ごとの個別運用ルールなどを含む実務完全再現  
  理由：本 PoC の目的は、まず実道路ネットワーク上で VRP の基本構造を成立させることにあり、個別業務ルールの完全反映はその後の詳細設計段階で扱うべきため。

- 倉庫内作業、積み込み時間、荷待ち時間を含む現場工程全体の最適化  
  理由：今回の対象は配送ルート最適化であり、倉庫内工程や荷役工程まで含めると問題設定が大きく変わるため、スコープを明確に分離する。

- 複数デポを前提とした拠点間配車や拠点配置の最適化  
  理由：本 PoC では、まず単一エリア・単純な配送体制での最適化成立性を確認することを優先しており、複数拠点の設計はモデル構造が別論点になるため。

- 当日の遅延、割込み配送、再配車を反映した動的再計画  
  理由：本 PoC は、前日に実行して翌日の配送計画案を出力する前提であり、運行中の再最適化までは対象に含めないため。

- 実際の配送実績データとの突合による運用品質検証  
  理由：本 PoC は、まず実道路ネットワークと最適化モデルの成立性を確認する段階であり、実績データに基づく運用品質評価は次段階の検証課題とするため。

- 法規制、拘束時間、休憩義務などの労務制約を厳密に反映した実装  
  理由：今回は配送計画の基本構造と道路ネットワーク利用可能性の確認を優先しており、法規・労務制約の厳密実装は実務適用段階で別途詳細設計が必要になるため。

- 本番運用を前提としたシステム実装、リアルタイム連携、継続的な自動更新  
  理由：本ポートフォリオは社内 PoC レベルの水準を目指した検証であり、業務システムとしての完成品実装ではなく、最適化基盤と設計妥当性の確認を目的としているため。

## 8. 成功条件

- OSM により都道府県単位の道路ネットワークを安定して取得できること
- 取得したネットワークを保存・再読込して再現性を保てること
- 県全体 routing が破綻せず、OD ベースで道のり距離を安定計算できること
- VRP の目的関数と制約条件が実務に即した前提として整理できること
- 後続の VRP 実装に向けて、できることとできないことを判断できること
