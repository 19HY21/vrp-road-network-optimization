# アルゴリズム設計

このドキュメントでは、`04_math_model.md` で定義した数理モデルを、どのソルバーを用いてどのように実装するかを定義する。

---

## 1. ドキュメントの目的

本ドキュメントは、`04_math_model.md` で定義した数理モデルを実装に落とし込むための設計判断を記録することを目的とする。

具体的には、ソルバーの選定根拠、処理の流れ、制約条件の実装方針、計算時間の制御方法、複数候補案の生成方法、実行不可能解の扱いを定義する。

本ドキュメントは数式の再定義を目的とせず、`04_math_model.md` で定義した内容を前提として、それをどのように実装するかの判断を記録する。

---

## 2. ソルバー選定

本 PoC では、ソルバーを理論比較だけで採択するのではなく、小規模テストケースを用いて各候補を実際に動かし、制約表現の成立と計算性能を実証した上で採用ソルバーを確定する。

なお、本章で挙げる候補ライブラリはすべて無償で利用可能であることを前提として選定している。

### 2.1 比較軸

以下の 5 軸を用いて候補ライブラリを比較する。

| 比較軸 | 選定理由 |
|---|---|
| VRP 専用機能の有無 | 部分巡回防止・デポ出発帰着・容量・時間窓が組み込みか手動実装かで実装コストが大きく異なる |
| アルゴリズムの種類 | 厳密解法（MIP）と近似解法（ヒューリスティクス）では解の性質と計算特性が根本的に異なり、採用判断に直結する |
| スケーラビリティ | 配送先件数の増加に対する計算時間の特性は、PoC から本番化への移行可否を判断する上で重要な指標となる |
| 保守運用性 | 本番化を見据えた場合、コミュニティ規模・ドキュメント品質・開発継続性が長期的な利用可否に影響する |
| 拡張性 | 本番化にあたり制約の追加・変更が発生することを想定し、対応しやすさを評価する |

### 2.2 候補ライブラリの概要とメリット・デメリット

#### OR-Tools Routing Library

Google が提供するオープンソース最適化ライブラリ OR-Tools に含まれる VRP 専用 API。
部分巡回防止・デポ出発帰着・容量制約・時間窓制約が組み込み機能として提供される。
初期解生成戦略と局所探索戦略を選択でき、計算時間と解の品質のバランスを調整できる。

| 比較軸 | 評価 | 補足 |
|---|---|---|
| VRP 専用機能の有無 | ✅ | 主要 VRP 制約が組み込み。部分巡回防止・ルート接続が自動処理される |
| アルゴリズムの種類 | 近似解法 | 局所探索・メタヒューリスティクスベース |
| スケーラビリティ | ✅ | ヒューリスティクスベースのため大規模でも実用的な計算時間で解が得られる |
| 保守運用性 | ✅ | Google が開発・維持。活発なコミュニティと充実したドキュメントを持つ |
| 拡張性 | ⚠️ | コールバックによる拡張は可能だが、VRP 専用 API の構造上、任意制約の追加は CP-SAT より制約される |

#### OR-Tools CP-SAT

同じ OR-Tools パッケージに含まれる汎用制約プログラミングソルバー。
VRP 専用機能は持たないため、部分巡回防止・ルート接続を含む全制約を手動で定式化する必要がある。
任意の制約を自由に表現できるため、カスタム制約への対応力は高い。

| 比較軸 | 評価 | 補足 |
|---|---|---|
| VRP 専用機能の有無 | ❌ | VRP 制約の全手動実装が必要 |
| アルゴリズムの種類 | 近似解法 | 制約プログラミング＋SAT ベース |
| スケーラビリティ | ⚠️ | VRP 専用の探索最適化がないため Routing Library より性能が劣る可能性がある |
| 保守運用性 | ✅ | Routing Library と同パッケージ。Google が維持 |
| 拡張性 | ✅ | 任意の制約を柔軟に追加・変更できる |

#### PuLP

Python 向けの線形計画・混合整数計画（MIP）モデリングライブラリ。バックエンドソルバーに CBC を標準利用する。
VRP を MIP として定式化するため、部分巡回防止には MTZ 制約等の手動実装が必要となる。

| 比較軸 | 評価 | 補足 |
|---|---|---|
| VRP 専用機能の有無 | ❌ | 部分巡回防止・ルート接続の手動実装が必要 |
| アルゴリズムの種類 | 厳密解法 | MIP・分枝限定法ベース |
| スケーラビリティ | ❌ | 配送先件数の増加により計算時間が指数的に悪化するリスクがある |
| 保守運用性 | ⚠️ | 小規模チームによるメンテナンス。コミュニティ規模は限定的 |
| 拡張性 | ✅ | 制約の追加・変更は容易 |

#### Pyomo

Python 向けの代数的最適化モデリング言語。学術・研究用途での実績が多く、CBC・GLPK 等のソルバーを別途インストールして利用する構成となる。
VRP 制約は手動実装が必要。

| 比較軸 | 評価 | 補足 |
|---|---|---|
| VRP 専用機能の有無 | ❌ | 部分巡回防止・ルート接続の手動実装が必要 |
| アルゴリズムの種類 | 厳密解法 | MIP ベース。ソルバーを別途設定して利用 |
| スケーラビリティ | ❌ | MIP ベースのため配送先件数増加で計算時間が悪化するリスクがある |
| 保守運用性 | ✅ | 学術コミュニティによる活発なメンテナンス。ドキュメントが充実している |
| 拡張性 | ✅ | 制約の追加・変更が容易で柔軟性が高い |

#### Python-MIP

CBC および Gurobi をバックエンドとして利用できる MIP モデリングライブラリ。PuLP に近い位置づけであり、VRP 専用機能は持たない。

| 比較軸 | 評価 | 補足 |
|---|---|---|
| VRP 専用機能の有無 | ❌ | 部分巡回防止・ルート接続の手動実装が必要 |
| アルゴリズムの種類 | 厳密解法 | MIP ベース |
| スケーラビリティ | ❌ | MIP ベースのため配送先件数増加で計算時間が悪化するリスクがある |
| 保守運用性 | ⚠️ | コミュニティ規模が小さく、開発活動が限定的 |
| 拡張性 | ✅ | 制約の追加・変更は容易 |

### 2.3 検証優先順位と採用方針

各候補を以下の優先順位で小規模テストケースに対して実際に動かし、本 PoC の全制約を表現できることと想定配送先件数での計算時間を確認した上で採用ソルバーを確定する。

| 優先順位 | ライブラリ | 優先する理由 |
|---|---|---|
| 1 | OR-Tools Routing Library | VRP 専用機能・スケーラビリティ・保守運用性の全軸で最有力 |
| 2 | OR-Tools CP-SAT | 同パッケージで学習コストが低く、拡張性で補完できる可能性がある |
| 3 | Python-MIP | MIP 系の中では構成がシンプルで検証コストが低い |
| 4 | PuLP | 保守運用性のリスクを踏まえ後順とする |
| 5 | Pyomo | ソルバー別途設定と構成の複雑さを考慮して最後順とする |

採用基準は以下とする。本 PoC の全制約を表現でき、想定配送先件数で許容計算時間内に解が得られることを確認できた最初のライブラリを採用する。

### 2.4 採用ソルバーの検証と選定根拠

#### テストケース定義

- 対象地点数: 30〜50 地点（神奈川県内の実座標を使用）
- 制約水準: `04_math_model.md` の第 7 章で定義した 11 の制約条件すべてを適用する
- デポ: 本 PoC で想定する単一デポを使用する

#### 検証方法

各候補ライブラリに対して同一テストケースを与え、探索時間の上限を統一した上で実行する。  
探索時間を統一することで、ライブラリの実装差による有利不利をなくし、公平な条件のもとで比較する。

#### 確認指標

| 指標 | 判定方法 |
|---|---|
| 解の有無 | ソルバーが実行可能解を返すかを自動判定する |
| 制約充足 | 全制約を満たしているかを自動判定する |
| 解の品質 | 統一探索時間内に得られた目的関数値（総コスト）を自動計測・比較する |
| 解の現実性 | 得られたルートを地図上にプロットし、人間が目視で経路の妥当性を確認する |

#### 採用基準

以下の優先順位で採用ソルバーを決定する。

1. 解が返ること、かつ全制約を満たしていること（必須条件。満たさない場合は失格とする）
2. 統一探索時間内の目的関数値が小さいこと（解の品質による優劣）
3. 目視確認により経路が現実的であること

必須条件を満たした候補が複数存在する場合は、解の品質と現実性確認の結果をもとに採用ソルバーを決定する。

#### 採用結果

検証完了後に以下の問いに答える形で記載する。

- 採用したソルバーは何か
- 採用した理由は何か（解の有無・制約充足・解の品質・現実性確認の結果をもとに）
- 他の候補を不採用とした理由は何か
- 目視確認の結果、経路に問題はなかったか
- 採用ソルバーで本 PoC の全制約を実装できると判断した根拠は何か

※ 検証完了後に記載する。

## 3. 処理フロー

本章では、入力データを受け取ってから出力成果物を格納するまでの処理ステップを定義する。各ステップの入力・出力および処理内容を明示する。

### 3.1 処理ステップ一覧

本 PoC では複数の候補案を生成するため、以下の 4 ステップを `vehicle_condition_master` で定義した稼働可能台数レンジ内の各台数値に対して 1 回ずつ実行する。1 回の実行が 1 候補案の生成に対応する。

| ステップ | 処理内容 | 入力 | 出力 |
|---|---|---|---|
| ① 入力データの読み込み | 前処理済みデータを読み込む | OD 行列・配送トランザクション・デポマスタ・時間帯マスタ | 有効配送先集合・距離・時間パラメータ |
| ② ソルバー入力の構築 | データとパラメータをソルバーに渡す形式に変換する | 有効配送先集合・OD 行列・車両条件・制約パラメータ | 制約・目的関数を含むソルバーモデル |
| ③ ソルバー実行 | 制約と目的関数をもとに最適化を実行する | ソルバーモデル・探索設定 | 最適化解（訪問順・使用台数・総走行距離） |
| ④ 出力変換・格納 | 解を出力形式に変換して書き出す | 最適化解 | candidate_plan_summary・candidate_route_detail |

### 3.2 各ステップの詳細

#### ① 入力データの読み込み

`02_data_definition.md` で定義した以下のデータを読み込み、数理モデルへ投入する入力を準備する。

- `od_matrix_result`: 地点対ごとの道のり距離と静的推定所要時間
- `delivery_transaction`: 有効配送先集合・需要量・時間帯区分
- `depot_master` / `vehicle_condition_master`: デポ情報・車両台数レンジ・車両容量・車両固定費・距離単価・業務時間
- `time_slot_master`: 時間帯区分ごとの許容時間帯

前処理段階で除外された失敗地点はこの時点で入力対象外として扱う。

ソルバー実行前に入力データのバリデーションを行い、問題が検出された場合（入力不備）はその台数設定をスキップし、発生した問題を記録する。詳細は第 7 章を参照。

#### ② ソルバー入力の構築

読み込んだデータをもとに、採用ソルバーのモデルを構築する。具体的には以下を行う。

- 地点集合（デポ＋有効配送先）の登録
- OD 行列を参照する距離コールバックの設定
- 目的関数（`車両固定費 × 使用台数 + 距離単価 × 総走行距離`）の定義
- 制約条件の設定（詳細は第 4 章で定義する）

#### ③ ソルバー実行

ソルバーは以下の手順で最適化を実行する。

1. **初期解の生成**: 初期解生成戦略に従い、制約を満たす初期ルートを生成する
2. **局所探索による解の改善**: 初期解をもとに局所探索戦略を用いて解を繰り返し改善する
3. **探索終了**: 設定した探索時間上限に達した時点で、それまでに得られた最良解を返す

#### 初期解生成戦略

以下の初期解生成戦略および局所探索戦略は OR-Tools Routing Library を対象とした内容である。MIP 系ライブラリ（PuLP・Pyomo・Python-MIP）を採用した場合、初期解生成および解の改善は分枝限定法の内部処理として扱われるため、これらの戦略選択は適用されない。採用ソルバーが MIP 系に確定した場合は本節を更新する。

初期解生成戦略とは、探索の出発点となる最初のルートをどのように構築するかの方針を指す。OR-Tools Routing Library では以下の戦略が利用可能である。

| 戦略 | 内容 |
|---|---|
| `AUTOMATIC` | OR-Tools が問題の構造に応じて戦略を自動選択する |
| `PATH_CHEAPEST_ARC` | デポから出発し、移動コストが最小の未訪問地点を順番につないでルートを構築する |
| `SAVINGS` | 各配送先を個別ルートとして出発し、統合することでコスト削減が最大となるルートを順次結合する（Clarke-Wright 法） |
| `CHRISTOFIDES` | Christofides アルゴリズムをベースに初期ルートを構築する。理論的に最適解の 1.5 倍以内の解が保証される |

初期解の品質は局所探索の出発点となるため、探索全体の効率に影響する。具体的な戦略の選定は採用ソルバー確定後に行う。

#### 局所探索戦略

局所探索戦略とは、初期解を出発点として解を繰り返し改善する方針を指す。単純に「改善する操作のみを受け入れる」方針では局所最適解に陥りやすいため、一時的な悪化を許容して局所最適解を脱出する戦略が VRP では有効である。OR-Tools Routing Library では以下の戦略が利用可能である。

| 戦略 | 内容 |
|---|---|
| `AUTOMATIC` | OR-Tools が問題の構造に応じて戦略を自動選択する |
| `GREEDY_DESCENT` | コストが改善する操作のみを受け入れる。高速だが局所最適解に陥りやすい |
| `GUIDED_LOCAL_SEARCH` | 頻繁に選ばれるルートにペナルティを課すことで探索の多様性を維持し、局所最適解を脱出する。VRP での有効性が高い |
| `SIMULATED_ANNEALING` | 探索初期には悪化する操作も確率的に受け入れ、徐々に受け入れ確率を下げる。局所最適解の脱出に有効 |
| `TABU_SEARCH` | 直近に訪れた解を禁断リストに登録して再訪を防ぎ、探索空間を広く探索する |

具体的な戦略の選定は採用ソルバー確定後に行う。

#### ④ 出力変換・格納

ソルバーが返した解を以下の出力形式に変換して書き出す。可視化成果物の生成はこのステップの対象外とする。

- `candidate_plan_summary`: 候補案単位の集計情報（使用台数・総走行距離・総概算コスト等）
- `candidate_route_detail`: 車両単位・訪問順単位の配送明細（配送先・区間距離・想定到着時刻等）

## 4. 制約条件の実装方針

`04_math_model.md` の第 7 章で定義した 11 の制約条件について、採用ライブラリによって自動処理されるものと開発者が手動実装するものが異なる。本章では OR-Tools Routing Library を例として切り分けを整理し、その他のライブラリを採用した場合の考え方を示す。

### 4.1 OR-Tools Routing Library を採用した場合の切り分け

OR-Tools Routing Library はルーティング問題に特化した設計であるため、一部の制約がライブラリの組み込み機能として自動処理される。

#### 自動処理される制約

| 制約条件 | 自動処理される理由 |
|---|---|
| 全件訪問制約 | ルーティングモデルに登録した全地点を必ず 1 回訪問する構造をモデルが保証する |
| デポ出発・デポ帰着制約 | モデル初期化時にデポを指定することで、全車両がデポを起点・終点とする構造が自動的に成立する |
| 車両使用とルート接続制約 | モデルがフロー保存則を内部で保証する。自己ループも許容されない |
| 配送先割当と地点間移動の整合制約 | ルーティングモデルの内部構造上、割当と実訪問ルートの整合性はモデルが保証する |
| 部分巡回防止制約 | ルーティングモデルが部分巡回を内部で排除する |
| 到着時刻伝播制約 | 時間ディメンションを追加することで、移動時間をもとにした到着時刻の伝播がモデル内部で自動処理される |

#### 開発者が手動実装する制約

| 制約条件 | 実装方針 |
|---|---|
| 車両容量制約 | 容量ディメンションを追加し、各配送先の需要量をコールバックで登録する。各車両の上限容量を設定することで制約を表現する |
| 稼働可能台数レンジ制約 | モデルに登録する車両数を上限値として設定し、最低使用台数を下限値として別途指定する |
| 業務開始時刻制約 | 時間ディメンションを追加し、デポ出発の時間窓を業務開始時刻に固定することで表現する |
| 業務時間内帰着制約 | デポへの帰着時刻の上限を業務終了時刻として時間ディメンションに設定することで表現する |
| 時間帯制約（午前・午後） | 各配送先の時間帯区分に対応する許容時間帯を、ノードごとの時間窓としてモデルに設定する。時間指定なしの配送先には時間窓制約を課さない |

### 4.2 その他のライブラリを採用した場合

OR-Tools CP-SAT・PuLP・Pyomo・Python-MIP を採用した場合、4.1 で自動処理とした制約を含む **11 制約すべてが開発者による手動実装の対象**となる。

特に部分巡回防止については、MIP ベースのライブラリでは MTZ 制約等を明示的に定式化する必要があり、実装コストが大きくなる。

採用ソルバーが確定した後、本章の実装方針を採用ライブラリに合わせて更新する。

### 4.3 PoC スコープにおける実装対象の範囲

`04_math_model.md` の第 7 章で定義した 11 の制約条件はすべて本 PoC の実装対象とする。

ただし、以下の点は本 PoC の前提として扱う。

- 時間帯制約で用いる所要時間は OSM 道路属性に基づく静的推定値であり、実交通を反映した動的時間ではない
- 荷下ろし時間・駐車待ち時間などの現場作業時間は扱わない
- 業務開始時刻・業務終了時刻は単一デポに紐づく共通条件として固定的に扱う

## 5. 計算時間制御

### 5.1 採用ライブラリ別の計算時間特性

候補ライブラリはアルゴリズムの種類によって計算時間の特性が異なる。本 PoC では以下の 2 種類を想定する。

#### ヒューリスティクス系（OR-Tools Routing Library・OR-Tools CP-SAT）

探索開始直後に大きな改善を見つけ、時間が経つにつれて改善幅が小さくなる傾向がある。一定時間以降は改善が頭打ちになるため、探索時間上限を設けることで実用的な解を効率よく得られる。

#### MIP 系（PuLP・Pyomo・Python-MIP）

分枝限定法をベースとするため、問題規模の増加に対して計算時間が指数的に増加するリスクがある。探索時間上限に達した場合は、それまでに見つかった最良の実行可能解を返す。最適性の証明まで到達できない場合でも解は得られるが、解の最適性は保証されない。

具体的な探索時間上限の設定は、第 2 章の検証を通じて採用ライブラリが確定した後に行う。

### 5.2 計算時間と解の品質のトレードオフ

いずれのライブラリでも、計算時間よりも**制約設計の質**が解の品質を支配する。制約が適切に定義されていれば探索空間が現実的な範囲に絞られ、短時間でも良質な解が得られる。逆に制約設計が不十分な場合、計算時間を延ばしても探索が無駄な空間を彷徨い続けるだけになる。

また、実行不可能解（infeasible）が発生している場合は、計算時間を延ばしても解は得られない。infeasible は制約設計の問題を示すフィードバックであり、時間制御ではなく制約の見直しで対処する（詳細は第 7 章を参照）。

### 5.3 計算時間延長の効果と限界

計算時間の延長が有効に機能するのは、以下の条件が揃っている場合に限られる。

- 制約が適切に定義されており、探索空間が現実的な範囲に絞られている
- 初期解生成後も探索が改善を続けられる余地がある

上記条件を満たさない場合、計算時間の延長による改善効果は限定的である。したがって、計算時間の設定は制約設計の妥当性を確認した後に行うことを前提とする。

なお、探索時間上限内に実行可能解が 1 件も得られなかった場合（時間内解なし）は、infeasible とは異なり解の存在が否定されているわけではない。まず探索時間を延ばした上で改善がなければ制約を見直す。詳細は第 7 章を参照。

### 5.4 検証方針（3 段階）

以下の 3 段階で探索時間上限を設定し、各段階での目的関数値を計測する。これにより、どの時間設定で解の改善が頭打ちになるかを実証的に確認する。

| 段階 | 探索時間上限 | 目的 |
|---|---|---|
| 短 | 30 秒 | 初期解品質と短時間での改善余地を確認する |
| 中 | 120 秒 | 解の品質と実行時間のバランスを確認する |
| 長 | 300 秒 | 計算時間を伸ばした際の改善幅の限界を確認する |

各段階で目的関数値（総コスト）を記録し、段階間の改善幅を比較する。改善幅が小さい場合は、その手前の時間設定を採用候補とする。

### 5.5 検証結果

検証完了後に以下の問いに答える形で記載する。

- 各段階の目的関数値はどうだったか
- 計算時間を延ばすにつれて解の改善は確認できたか
- 改善が頭打ちになった段階はどこか
- 採用する探索時間上限はどれか、またその根拠は何か

※ 検証完了後に記載する。

## 6. 複数候補案の生成

本章では、`04_math_model.md` の第 9.2 節で定義した複数候補案を、アルゴリズム上どのように生成するかを定義する。

### 6.1 複数候補案を提示する目的

配送計画の最終採用判断は現場担当者および責任者が行う。数理最適化が返す解は目的関数上の最適解であり、業務上の都合・現場の制約・コスト方針によって採用されない場合がある。その際、単一解しか持たない場合は条件を変えた再実行が必要となり、現場の意思決定が滞る。

本 PoC では、稼働可能台数レンジ内の各台数設定に対して最適化を実行し、異なる使用台数のもとで得られた複数の実行可能解を候補案として事前に提示する。これにより、現場担当者および責任者が最適化済みの選択肢の中から業務判断を加えて採用案を決定できる。

### 6.2 生成パターン

候補案の違いはユーザーが設定するパラメータの違いに基づく。本 PoC では使用台数を生成パターンの軸とする。使用台数が変わると最適化によって得られるルート構成も変わるため、各候補案は台数とルート構成の両方で異なる案として区別される。実行ロジックの詳細は第 3 章を参照。

### 6.3 提示件数

生成される候補案の件数は稼働可能台数レンジの幅によって決まる。比較対象として目的関数値の小さい上位 5 件を提示することを基本とする。レンジ幅が 5 件未満の場合は全件を提示する。

### 6.4 推奨順位

候補案には以下の優先順位で推奨順位を付与し、`candidate_plan_summary` の `rank` として出力する。

1. 使用台数が少ない案を上位とする
2. 使用台数が同一の場合は総走行距離が短い案を上位とする

この順位は目的関数の評価軸（使用台数削減優先・同台数内での走行距離最小化）に対応する。推奨順位はシステムによる自動評価であり、最終採用の決定は利用者が行う。

### 6.5 利用者別の選定観点

推奨順位はシステムが目的関数に基づいて自動付与するものであるが、採用案の決定には業務上の判断が伴う。利用者ごとの主な選定観点を以下に示す。

#### 配送計画担当者

- 各車両の担当範囲・荷物数配分に運用上許容できない偏りがないか
- 使用台数と総走行距離のバランスが実際の運用体制と合致しているか
- 総概算コストが業務上の許容範囲に収まっているか

#### 現場責任者および導入判断者

- 総概算コストおよび固定費・変動費の内訳が導入判断の根拠として妥当か
- 全制約条件が充足されているか
- 使用台数の削減によるスケジュール余裕の低下が現場リスクとして許容できるか

## 7. 実行不可能解の処理方針

本章では、ソルバーが解を返せない場合に発生しうる異常状態の定義・検知方法・要因と対処方針・出力方針を定義する。

### 7.1 異常状態の定義と区別

ソルバーが解を返せない状態は以下の 3 種類に区別する。

| 状態 | 定義 |
|---|---|
| 実行不可能解（infeasible） | ソルバーが現行の制約条件のもとで実行可能解が存在しないと判定した状態 |
| 時間内解なし | 探索時間上限内に実行可能解が 1 件も得られなかった状態。解の存在は否定されていない |
| 入力不備 | ソルバーへの入力データに問題があり、最適化を開始できない状態 |

実行不可能解と時間内解なしは異なる状態であり、対処方針も異なる。実行不可能解は計算時間を延ばしても解決しない（第 5.2 節参照）。時間内解なしは探索時間の延長または制約の見直しによって改善できる可能性がある。

### 7.2 各状態の検知方法

ソルバー実行後に返却されるステータスをもとに状態を判定する。

| 状態 | OR-Tools Routing Library | MIP ベースのライブラリ |
|---|---|---|
| 実行不可能解 | `routing.status()` が `ROUTING_INFEASIBLE` を返す | ソルバーステータスが `INFEASIBLE` を返す |
| 時間内解なし | `routing.status()` が `ROUTING_FAIL` を返す、または解オブジェクトが空 | ソルバーステータスが `TIME_LIMIT` かつ解なし |
| 入力不備 | ソルバー実行前のバリデーションで検出 | ソルバー実行前のバリデーションで検出 |

### 7.3 実行不可能解の要因と対処方針

実行不可能解が発生する主な要因と、解が得られる状態にするための対処方針を以下に示す。

| 要因 | 対処方針 |
|---|---|
| 稼働可能台数レンジ内の最大台数でも全件訪問が成立しない | `vehicle_condition_master` の台数上限を引き上げる |
| 車両容量に対して需要量の合計が超過している | 車両容量の上限を引き上げる、または需要量データを見直す |
| 配送件数・移動距離に対して業務時間が不足している | 業務終了時刻を延ばす、または配送件数を分割して実行する |
| 時間帯制約と移動時間の組み合わせが成立しない | 時間帯区分の許容時間帯を緩和する、または時間帯指定のある配送先の設定を見直す |

### 7.4 出力方針

第 3 章で定義した台数ごとの実行において、いずれかの台数設定で異常状態が発生した場合はその台数設定をスキップし、次の台数設定の実行へ進む。

スキップした台数設定については、通常の候補案（`candidate_plan_summary` / `candidate_route_detail`）は生成せず、以下の情報を記録する。

- 発生した状態の種別（実行不可能解・時間内解なし・入力不備）
- 判定可能な場合は 7.3 に示した要因のうち該当するものの候補

なお、ソルバーが返すステータスだけでは要因を一意に特定できない場合がある。その場合は要因候補を列挙する形で記録する。出力形式の詳細は `06_experiment_design.md` で定める。
