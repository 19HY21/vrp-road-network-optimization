# 数理モデル設計

このドキュメントでは、`00_problem_definition.md`、`01_business_requirements.md`、`02_data_definition.md`、`03_network_design.md` で整理した内容をもとに、VRP PoC で採用する数理最適化モデルを定義する。

本ドキュメントでは、単に目的関数や制約条件を列挙するのではなく、業務要件として何を守るべきか、道路ネットワーク計算結果をどのように数理モデルへ接続するか、現場利用に耐える現実的な解をどのように担保するかを整理対象とする。

## 1. ドキュメントの目的

本ドキュメントの目的は、`00_problem_definition.md`、`01_business_requirements.md`、`02_data_definition.md`、`03_network_design.md` で整理した内容をもとに、VRP PoC で採用する数理最適化モデルを定義することである。

本ドキュメントでは、業務要件、入力データ、道路ネットワーク計算結果を、どのような最適化問題として定式化するかを整理する。  
これにより、本 PoC がどのような入力を受け取り、何を意思決定し、どの制約条件のもとで、どのような結果を返すのかを一通り理解できる状態を目指す。

また、本 PoC では、単に数学的に解けることではなく、現場で利用可能な現実的な解を返すことを重視する。  
そのため、本ドキュメントでは、目的関数、入力データ、意思決定変数、制約条件、出力条件に加えて、業務上不自然な解を避けるためにモデルへ組み込むべき条件も整理対象とする。

## 2. 数理モデル化の基本方針

本 PoC では、単一都道府県内配送を前提とし、現時点では神奈川県内の単一デポを対象とする。  
各車両は単一デポから出発し、配送完了後は同じデポへ帰着する closed VRP として扱う。

配送対象は、入力時点で指定された配送先全件を原則対象とする。  
ただし、住所不備、ジオコーディング失敗、道路ネットワーク割当失敗、OD 計算失敗などにより、計算対象として投入できない地点がある場合は、その地点を除いた有効配送先集合を数理モデルの入力とする。  
したがって、本 PoC の数理モデルは未訪問を許容する設計ではなく、モデルに投入された有効配送先集合に対して全件訪問を課す前提とする。

使用台数は固定値ではなく、単一デポに紐づく稼働可能台数の下限値と上限値の範囲内で決定する。  
そのうえで、各配送先をどの車両が担当するか、どの順序で訪問するか、どの車両を使用するかを数理最適化によって決定する。

本 PoC の目的は、`車両固定費 × 使用台数 + 距離単価 × 総走行距離` で表現する総コストを最小化することである。  
ただし、業務上は固定費寄与が大きい前提のため、実質的には使用台数削減を強く優先し、そのうえで総走行距離短縮を第二の改善軸として扱う。

配送ルートの距離および所要時間は、OpenStreetMap を正本とする道路ネットワーク計算結果を用いる。  
これにより、直線距離ではなく、道路接続、一方通行、通行可否を踏まえた実道路ベースの配送ルートを扱う。

一方で、所要時間は OSM 道路属性に基づく静的推定値であり、実交通を反映した実時間ではない。  
そのため、他の地図サービスと距離や所要時間に差が生じる可能性があることを前提とし、所要時間は主目的関数ではなく、業務時間内帰着および `午前 / 午後 / 時間指定なし` の時間帯制約確認のための制約または参考値として扱う。

また、本 PoC では、単に数学的に実行可能な解を返すことではなく、現場で実用性のある解を返すことを重視する。  
そのため、実道路距離、静的推定時間、業務時間、時間帯制約を用いて、業務上不自然な解を抑制する設計とする。  
ただし、PoC の出力は最終判断を完全自動化するものではなく、採用可否の最終判断は配送計画担当者および現場責任者が行う前提とする。

## 3. 問題設定

本 PoC では、単一デポ、配送先集合、車両集合から構成される配送計画問題を対象とし、業務条件を満たしながら総コストを最小化する問題として扱う。

対象となる配送先は、入力時点で指定された配送先集合のうち、前処理および道路ネットワーク計算が成立した有効配送先集合とする。  
これらの配送先は、単一デポから出発する車両によって当日中に配送され、各車両は配送完了後に同じデポへ帰着するものとする。

各配送先には荷物個数ベースの需要量を持たせ、各車両には積載上限を持たせる。  
また、使用可能な車両台数はデポごとに下限値と上限値を持ち、その範囲内で使用台数を決定する。

本 PoC では `台数 = 人数` を前提とし、1 台あたり 1 人が配送を担当するものとする。  
さらに、各車両は業務開始時刻を起点にデポから出発し、業務終了時刻までにデポへ帰着することを前提とする。

本 PoC では、車両ごとの容量差や固定費差は設けず、車両は同一仕様として扱う。  
したがって、車両容量、固定費、距離比例費などの主要パラメータは、単一デポに紐づく共通条件として定義する。

距離および所要時間は、OpenStreetMap を正本とする道路ネットワーク計算結果を用いる。  
距離は実道路ネットワーク上の最短道のり距離とし、最短時間経路に基づく距離は用いない。所要時間は OSM 道路属性に基づく静的推定値とする。距離計算の詳細は `03_network_design.md` の第 8 章を参照。

また、本 PoC では、配送先ごとに `午前 / 午後 / 時間指定なし` の時間帯区分を持たせる。  
そのため、配送計画は、全件訪問、車両容量、稼働可能台数レンジ、業務時間内帰着、時間帯制約を同時に満たす問題として定義する。

一方で、本 PoC では、荷下ろし時間、駐車時間、顧客ごとの個別対応時間などの詳細な現場作業時間は、現時点では数理モデルの対象外とする。  
したがって、業務時間内帰着および時間帯制約の判定は、主として道路ネットワーク上の移動時間をもとに扱う。

なお、本 PoC では、原則として入力時点で指定された配送先全件を配送対象とする。  
ただし、住所不備、ジオコーディング失敗、道路ネットワーク割当失敗、OD 計算失敗などにより、計算対象として投入できない地点がある場合は、その地点を除いた有効配送先集合を対象として最適化を実行する。  
したがって、本章で定義する問題設定は、モデルに投入された有効配送先集合に対して成立するものとする。

## 4. 目的関数

本 PoC の目的関数は、配送計画に要する総コストを最小化することとする。  
総コストは、車両固定費と距離比例費に分けて簡略化し、以下で表現する。

`総コスト = 車両固定費 × 使用台数 + 距離単価 × 総走行距離`

ここで、車両固定費は、1 台の車両を稼働させるごとに発生する人件費相当および車両稼働固定費を簡略化したパラメータとして扱う。  
また、距離単価は、燃料費等の距離比例費を簡略化したパラメータとして扱う。

本 PoC では、`1台 = 1人` を前提としているため、使用台数が 1 台増えるごとに 1 人分の人件費相当の固定費が追加で発生する。  
そのため、総走行距離の短縮による距離比例費の削減よりも、使用台数削減による固定費削減の方がコストへの影響が大きい前提を置く。

したがって、目的関数は総コスト最小化として表現する一方で、解の評価方針としては使用台数削減を強く優先し、同一台数内で総走行距離短縮を図る考え方を採る。

また、総走行距離を短縮することには、距離比例費を抑える効果がある。  
そのため、使用台数が同一である候補案同士を比較する場合は、総走行距離が短い案ほど、変動費を抑えた解として評価する。

なお、ここで定義する目的関数は、数理最適化モデル内部で最小化する評価基準であり、出力時の候補案順位付けとは役割を分けて扱う。  
候補案順位付けは、業務上比較しやすい順序で提示するための補助的な評価ルールとして別途扱う。

## 5. 入力データとパラメータ

本章では、数理最適化モデルへ投入する入力データとパラメータを定義する。  
本 PoC では、住所正規化、ジオコーディング、道路ネットワーク割当、OD 計算は数理モデルの前段で実施済みであることを前提とし、数理モデルはそれらの前処理結果を受け取って最適化を行う。

数理モデルへ投入する入力は、大きく以下の 4 系統に分ける。

- 配送トランザクション由来の入力
- デポおよび車両条件マスタ由来の入力
- 時間帯マスタ由来の入力
- 道路ネットワーク計算結果由来の入力

### 5.1 配送トランザクション由来の入力

配送トランザクション由来の入力は、当日の配送対象を定義するために用いる。  
具体的には、`delivery_transaction` から、対象配送先、需要量、配送時間帯区分などを取得し、数理モデルの入力とする。

数理モデルへ渡す主な項目は以下のとおりとする。

- 配送対象となる有効配送先集合
- 各配送先の需要量
- 各配送先の時間帯区分
- 各配送先に対応する道路ネットワーク参照点

ここでいう有効配送先集合とは、入力時点で指定された配送先集合のうち、前処理および道路ネットワーク計算が成立した配送先のみで構成される集合である。  
住所不備、ジオコーディング失敗、道路ネットワーク割当失敗、OD 計算失敗などにより計算対象へ投入できない地点は、数理モデルの入力対象から除外する。

### 5.2 デポおよび車両条件マスタ由来の入力

デポおよび車両条件マスタ由来の入力は、配送計画の基準条件を定義するために用いる。  
本 PoC では単一デポを前提とするため、対象デポは 1 件に固定される。

数理モデルへ渡す主な項目は以下のとおりとする。

- 対象デポ
- デポの道路ネットワーク参照点
- 使用可能な車両台数の下限値
- 使用可能な車両台数の上限値
- 車両容量
- 車両固定費
- 距離単価
- 業務開始時刻
- 業務終了時刻

本 PoC では、車両は同一仕様として扱うため、車両容量、車両固定費、距離単価は単一デポに紐づく共通パラメータとして扱う。  
また、`1台 = 1人` を前提とするため、使用台数はそのまま必要人員数に対応する。

### 5.3 時間帯マスタ由来の入力

時間帯マスタ由来の入力は、配送先ごとの時間帯制約を判定するために用いる。  
本 PoC では、配送先ごとに `午前 / 午後 / 時間指定なし` の区分を持たせ、必要に応じて各区分に対応する許容時間帯を `time_slot_master` から参照する。

数理モデルへ渡す主な項目は以下のとおりとする。

- 時間帯区分コード
- 各時間帯区分に対応する開始時刻
- 各時間帯区分に対応する終了時刻

これにより、配送先ごとの時間帯要件を、数理モデル上の時間制約として扱えるようにする。

### 5.4 道路ネットワーク計算結果由来の入力

道路ネットワーク計算結果由来の入力は、配送先間およびデポとの移動距離・移動時間を数理モデルへ渡すために用いる。  
本 PoC では、距離および所要時間は数理モデル内部で再計算せず、`od_matrix_result` に保持した計算結果を入力パラメータとして利用する。

数理モデルへ渡す主な項目は以下のとおりとする。

- 地点対ごとの道のり距離
- 地点対ごとの静的推定所要時間

ここで扱う地点対は、デポから配送先、配送先からデポ、配送先から配送先の有向組み合わせとする。  
また、所要時間は実交通を反映した実時間ではなく、OSM 道路属性および補正条件に基づく静的推定値として扱う。

### 5.5 数理モデル上の集合とパラメータの考え方

本 PoC では、前述の入力データを数理モデル上で集合およびパラメータとして整理して扱う。  
代表的な考え方は以下のとおりとする。

- 配送先集合: 有効配送先集合
- 車両集合: 使用可能台数上限にもとづいて定義する車両候補集合
- 地点集合: デポと有効配送先集合を合わせた集合
- 需要量パラメータ: 各配送先に対応する荷物個数
- 容量パラメータ: 各車両に共通の積載上限
- 距離パラメータ: 地点対ごとの道のり距離
- 時間パラメータ: 地点対ごとの静的推定所要時間
- コストパラメータ: 車両固定費および距離単価
- 時間制約パラメータ: 業務開始時刻、業務終了時刻、時間帯ごとの許容時間帯

なお、失敗地点および失敗地点対は、この章で定義する入力パラメータの生成前に除外または別管理されるため、数理モデル内部では扱わないものとする。

## 6. 意思決定変数

本章では、数理最適化モデルが解として何を決定するかを定義する。  
本 PoC では、単に総コストを算出するのではなく、使用台数、各配送先の担当車両、各車両の移動経路、訪問順序、到着時刻を意思決定対象とする。

本 PoC では、車両はすべて同一仕様とし、数理モデル上は `k = 1, 2, ..., K` の連番で管理する。  
ここで `K` は、対象デポに対して使用可能な車両台数上限にもとづいて定義する車両候補数とする。

### 6.1 車両使用有無を表す変数

各車両候補 `k` を実際に使用するかどうかを表す変数を定義する。  
この変数は、指定された台数レンジの範囲内で実際の使用台数を決定するために用いる。

また、この変数は、使用台数の算出、車両固定費の計算、台数レンジ制約との接続に利用する。

### 6.2 配送先割当を表す変数

各配送先 `i` を、どの車両 `k` が担当するかを表す変数を定義する。  
この変数は、各配送先がどの車両によって配送されるかを明示的に表現するために用いる。

本 PoC では、配送先割当を地点間移動変数から暗黙的に導出するのではなく、独立した意思決定変数として保持する。  
これにより、全件訪問制約、車両容量制約、配送先担当関係を設計書上で明確に扱えるようにする。

### 6.3 地点間移動を表す変数

各車両 `k` が、地点 `i` から地点 `j` へ移動するかどうかを表す変数を定義する。  
ここで地点とは、単一デポおよび有効配送先集合から構成される地点集合を指す。

この変数は、各車両の配送ルートを構成する中心的な変数であり、デポ出発・デポ帰着制約、ルート接続制約、部分巡回防止制約、総走行距離計算、総移動時間計算と接続するために用いる。

### 6.4 到着時刻を表す変数

各車両 `k` が各地点 `i` に到着する時刻を表す変数を定義する。  
この変数は、地点間走行時間パラメータをもとに、各配送先への到着時刻および帰着時刻を表現するために用いる。

本 PoC では、地点間の走行時間そのものは `od_matrix_result` に保持した入力パラメータとして扱い、数理モデル内部では再計算しない。  
一方で、業務時間内帰着制約および時間帯制約を扱うために、到着時刻を意思決定変数として保持する。

### 6.5 意思決定変数と出力の関係

本章で定義する意思決定変数は、数理モデル内部で最適化対象を表現するためのものである。  
現場利用者や導入判断者に提示する配送ルート、使用台数、総走行距離、総移動時間、配送コストは、これらの意思決定変数から導出される出力情報として、後続の出力条件で整理する。

## 7. 制約条件

本章では、業務上守るべき条件を数理モデルの制約として定義する。  
配送の基本構造に関わる制約から、現場利用に必要な現実的制約までを順に整理する。  
本 PoC では、前処理段階で有効配送先集合および地点対距離・地点対所要時間が確定している前提で、数理モデル内部では当該集合に対して実行可能な配送計画のみを許容する。

### 7.1 全件訪問制約

有効配送先集合に含まれる全配送先は、必ず 1 回だけ訪問対象とする。  
したがって、同一配送先への重複訪問および未訪問は、ともに許容しない。

また、本制約は、数理モデルへ投入された有効配送先集合に対して適用する。  
前処理段階で除外された失敗地点は、この制約の対象外であり、数理モデル内部で未訪問ペナルティ等により扱うことはしない。

### 7.2 デポ出発・デポ帰着制約

各使用車両は、単一デポから 1 回だけ出発し、配送完了後に同じデポへ 1 回だけ帰着するものとする。  
これにより、本 PoC の数理モデルは closed VRP としての基本構造を満たす。

一方で、使用しない車両候補については、デポ出発およびデポ帰着を持たせない。  
そのため、デポ出発・デポ帰着は、車両使用有無変数と整合した形でのみ成立するものとする。

### 7.3 車両使用とルート接続制約

各車両の移動は、当該車両が使用される場合にのみ許容する。  
したがって、使用していない車両に対して地点間移動を持たせることは認めない。

また、各車両について、ある配送先へ流入した場合は同じ車両で当該配送先から流出するものとし、ルートの接続が保たれる状態を要求する。  
これにより、移動変数が断片的に存在する状態や、訪問後にルートが途切れる状態を防ぐ。

さらに、同一地点から同一地点への移動、すなわち `i = j` の自己ループは許容しない。  
これには `デポ -> デポ` および `配送先 -> 同一配送先` の両方を含む。

### 7.4 配送先割当と地点間移動の整合制約

各配送先について、割り当てられた車両は、その配送先を実際に訪問するルートを持たなければならない。  
したがって、ある配送先を車両 `k` が担当すると定義した場合、当該車両はその配送先へ流入し、かつその配送先から流出する移動を持つものとする。

逆に、配送先を担当していない車両が、当該配送先へ流入または流出することは認めない。  
これにより、配送先割当変数と地点間移動変数の意味が一致し、割当だけが存在して実訪問が存在しない状態、または訪問しているのに担当車両が未定義である状態を防ぐ。

### 7.5 車両容量制約

各車両に割り当てられた配送先の需要量合計は、当該車両の容量を超えてはならない。  
本 PoC では、需要量および車両容量はともに荷物個数ベースで扱うため、容量制約も荷物個数の総和に対して適用する。

これにより、単一車両へ過剰な配送先を割り当てることを防ぎ、現場で積載不可能な解を排除する。

### 7.6 稼働可能台数レンジ制約

使用台数は、対象デポに設定された下限値以上かつ上限値以下でなければならない。  
これにより、当日の業務条件として与えられた台数レンジを数理モデルへ明示的に反映する。

本制約により、必要以上に多くの車両を使用する解や、必要最低台数を下回る解を除外する。  
また、`1台 = 1人` の前提を置くため、本制約は必要人員数の範囲制約としても機能する。

### 7.7 部分巡回防止制約

ルート全体は単一デポと接続していなければならず、デポと無関係な閉路、すなわち部分巡回を許容しない。  
これにより、配送先同士だけで閉じた巡回が生成されることを防ぎ、各車両のルートがデポ出発・デポ帰着の構造を持つことを担保する。

本制約は、ルート接続制約だけでは防ぎきれない孤立巡回を排除するために必要とする。

### 7.8 業務開始時刻制約

各使用車両は、業務開始時刻を起点としてデポから出発するものとする。  
本 PoC では簡略化のため、使用車両は業務開始時刻より前に出発しないことに加え、業務開始時刻を基準として配送を開始する前提を置く。

これにより、業務開始前の先行出発によって実際には不可能な配送計画が成立することを防ぐ。

### 7.9 業務時間内帰着制約

各使用車両は、最終訪問後にデポへ帰着した時刻が業務終了時刻以下でなければならない。  
これにより、当日中に配送を完了し、所定の業務時間内に帰着できる解のみを許容する。

本制約は、静的推定所要時間を用いて、極端に長いルートや時間無視の非現実解を排除するために用いる。  
ただし、用いる時間は実交通を反映した実時間ではなく、OSM 道路属性に基づく静的推定値である。

### 7.10 時間帯制約

各配送先の到着時刻は、当該配送先に指定された時間帯区分に対応する許容時間帯を満たさなければならない。  
本 PoC では、`午前 / 午後 / 時間指定なし` を対象とし、`時間指定なし` については明示的な時間帯制約を課さない。

時間帯整合の判定には、地点対所要時間および到着時刻変数を用いる。  
ただし、ここで扱う所要時間は静的推定値であり、厳密な到着保証を目的とするものではなく、業務上不自然な解を抑制するための制約として扱う。

### 7.11 到着時刻伝播制約

ある車両が地点 `i` から地点 `j` へ移動する場合、地点 `j` の到着時刻は、地点 `i` の到着時刻に地点対所要時間を加えた値以上でなければならない。  
これにより、地点間移動変数と到着時刻変数の間に一貫した時間関係を持たせる。

本 PoC では、荷下ろし時間や駐車時間などの詳細作業時間を現時点で扱わないため、到着時刻の伝播は主として地点間の移動時間にもとづいて定義する。  
また、本制約により、業務開始時刻制約、業務時間内帰着制約、時間帯制約が、実際のルートと接続した形で評価されるようにする。

## 8. 現実性担保と PoC スコープ外事項

本章では、本 PoC が現場で利用可能な現実的な解をどのように担保しているか、および現時点で PoC の対象外として切り分けている事項を整理する。  
ここで扱う内容は、新たな制約条件を追加定義するものではなく、これまでに定義した制約条件および対象範囲の前提を、現実性の観点から再整理するものである。

### 8.1 制約条件によって担保する現実性

本 PoC では、数理モデルに以下の制約条件を設けることで、業務上不自然な解を抑制する。

- 全件訪問制約により、有効配送先集合に含まれる配送先の未訪問および重複訪問を防ぐ
- デポ出発・デポ帰着制約により、各車両が単一デポを起点および終点とする closed VRP の構造を満たす
- 車両使用とルート接続制約により、未使用車両の移動や自己ループを防ぎ、連続したルートのみを許容する
- 配送先割当と地点間移動の整合制約により、担当車両と実際の訪問ルートが一致する状態を保つ
- 車両容量制約により、1 台に過剰な荷物を割り当てる解を防ぐ
- 稼働可能台数レンジ制約により、現場で使用可能な人数・車両数の範囲を超える解を防ぐ
- 部分巡回防止制約により、デポと無関係な閉路が発生する解を防ぐ
- 業務開始時刻制約および業務時間内帰着制約により、当日の業務時間を超える解を防ぐ
- 時間帯制約により、`午前 / 午後 / 時間指定なし` の配送条件に反する解を防ぐ
- 到着時刻伝播制約により、移動時間と到着時刻の関係が保たれたルートのみを許容する

また、本 PoC では、距離および所要時間として OpenStreetMap を正本とする道路ネットワーク計算結果を利用する。  
これにより、直線距離や道路接続を無視した簡易な解ではなく、一方通行、通行可否、道路接続を踏まえた実道路ベースの配送計画を扱う。

### 8.2 PoC のスコープ外として扱う事項

一方で、本 PoC は現場運用を完全再現することを目的とした本番システムではなく、実道路ネットワークベース VRP の有効性を検証する PoC である。  
そのため、以下の事項は現時点では数理モデルの対象外、または PoC の対象範囲外として扱う。

- 渋滞、事故、時間帯混雑を反映した動的交通情報
- 荷下ろし時間、駐車待ち時間、顧客対応時間などの詳細な現場作業時間
- 再配達や当日中の配送先変更などの動的な業務変更
- 配送員ごとの熟練差や運転特性差
- 複数デポを前提とする配送計画
- 都道府県を跨ぐ広域配送
- 徒歩や他交通手段を組み合わせた複合移動
- 分単位の厳密な到着保証を要する運用

これらの事項は、現場運用上は重要である一方で、本 PoC の目的である「単一デポ・実道路ネットワークベースの VRP による配送計画最適化」の検証範囲を超えるため、現時点では扱わない。


## 9. 実行可能性と暫定結果の扱い

本章では、数理モデル上の実行可能解、業務上比較対象となる候補案、前処理不備を含む暫定結果、および実行可能解なしの状態を整理する。  
ここで重要なのは、候補案と暫定結果を同一視しないことである。

### 9.1 数理モデル上の実行可能解

本 PoC における数理モデル上の実行可能解とは、`7. 制約条件` で定義した制約をすべて満たす解を指す。  
具体的には、有効配送先集合に対する全件訪問、単一デポ出発・帰着、車両容量、稼働可能台数レンジ、業務開始時刻、業務時間内帰着、時間帯制約などをすべて満たすことを条件とする。

したがって、数理モデル上の実行可能解は、モデル内部で正規に成立した解であり、この時点では暫定結果ではない。

### 9.2 候補案の位置づけ

本 PoC では、数理モデル上の実行可能解の中から、使用台数、総走行距離、総コスト、制約充足状況などを比較可能な形で複数案提示する。  
これらは、配送計画担当者および導入判断者が比較・検討するための候補案であり、正規の比較案として扱う。

候補案は、数理モデル上の実行可能解を業務判断しやすい形で提示したものであり、候補案であること自体をもって暫定結果とはしない。  
最終的な採用案の決定は、PoC の出力をもとに現場担当者および現場責任者が行う。

### 9.3 暫定結果の位置づけ

本 PoC における暫定結果とは、前処理段階または道路ネットワーク計算段階で一部地点に失敗が発生し、入力時点で指定された配送先全件を数理モデルへ投入できなかった状態で得られた結果を指す。

具体的には、以下のような場合が該当する。

- 住所不備
- ジオコーディング失敗
- 道路ネットワーク割当失敗
- OD 計算失敗

この場合、失敗地点または失敗地点対を除いた有効配送先集合に対して数理最適化を実行するため、数理モデル上は実行可能解であっても、業務上の原則である全件配送前提を満たしていない。  
そのため、当該結果は確定評価結果ではなく、修正および再実行を前提とした暫定結果として扱う。

### 9.4 実行可能解なしの扱い

一方で、前処理不備とは別に、数理モデルへ投入した有効配送先集合に対して制約条件を満たす解が存在しない場合がある。  
これは暫定結果とは別の状態であり、`実行可能解なし` として独立に扱う。

想定される要因は以下のとおりである。

- 稼働可能台数レンジでは全件配送が成立しない
- 車両容量制約により全件配送が成立しない
- 業務時間内帰着制約により成立しない
- 時間帯制約により成立しない
- raw データの条件が本 PoC の前提と整合していない

この状態は、候補案の不採用や暫定結果とは異なり、現行条件では数理モデル上の解が存在しないことを意味する。  
そのため、制約条件、台数レンジ、業務条件、入力データの妥当性を見直す対象として扱う。

### 9.5 本章における整理の考え方

以上を踏まえ、本 PoC では以下のように整理する。

- 制約条件を満たした解: 数理モデル上の実行可能解
- 実行可能解を比較用に提示したもの: 候補案
- 一部失敗地点を除いて得た結果: 暫定結果
- 制約条件を満たす解が存在しない状態: 実行可能解なし

この整理により、候補案、暫定結果、実行不能状態を混同せず、業務上の判断とデータ修正対応を分けて扱えるようにする。

## 10. 出力条件

本章では、数理最適化モデルの解から、後続の業務利用および比較評価に必要な出力情報をどのような粒度で返すかを定義する。  
本 PoC では、出力を単一の結果表示にまとめるのではなく、候補案単位、ルート明細単位、可視化単位に分けて保持し、利用者ごとに必要な情報を参照できるようにする。

また、出力は、現場利用者向けの実行情報、配送計画担当者向けの比較情報、現場責任者および導入判断者向けの評価情報を含むものとする。

### 10.1 候補案単位で返す情報

候補案単位の出力は、配送計画担当者および現場責任者が複数案を比較するために用いる。  
本 PoC では、数理モデル上の実行可能解を候補案として整理し、`candidate_plan_summary` に相当する粒度で保持する。

候補案単位で返す主な情報は以下のとおりとする。

- `candidate_plan_id`
- 使用台数
- 総走行距離
- 総概算コスト
- 固定費部分
- 距離比例費部分
- 総訪問件数
- 総荷物個数
- 業務時間内帰着可否
- 時間帯制約充足可否
- 実行可能性フラグ
- システム推奨案フラグ
- 推奨順位
- 推奨理由
- 目的関数値
- 生成日時

これにより、使用台数、距離、コスト、制約充足状況を横並びで比較できる状態を作る。

### 10.2 ルート明細単位で返す情報

ルート明細単位の出力は、現場利用者および配送計画担当者が、各車両ごとの配送順序や担当範囲を具体的に確認するために用いる。  
本 PoC では、`candidate_route_detail` に相当する粒度で、車両単位および訪問順単位の情報を保持する。

ルート明細単位で返す主な情報は以下のとおりとする。

- `candidate_plan_id`
- 車両識別子
- 訪問順
- 出発地点および帰着地点
- 配送先識別子
- 配送先住所
- 当該配送先の荷物数
- 当該車両の累積荷物数
- 区間距離
- ルート総距離
- 累積距離
- 区間所要時間
- 累積所要時間
- 指定時間帯区分
- 計画上の想定到着時刻
- 計画上の想定出発時刻
- 計画上の想定時間帯区分

これにより、各車両がどの配送先をどの順序で担当し、どの程度の荷物数と移動負荷を持つかを確認できるようにする。

### 10.3 利用者視点ごとの主な確認対象

本 PoC の出力は、利用者ごとに主に確認する観点が異なることを前提とする。

- 実配送担当者  
  各車両の配送ルート、訪問順、担当配送先、想定到着順を主に確認する

- 配送計画担当者  
  使用台数、各車両の担当範囲、荷物数配分、総距離、候補案比較結果を主に確認する

- 現場責任者および導入判断者  
  総概算コスト、固定費と距離比例費の内訳、制約充足状況、候補案比較結果を主に確認する

このように、同一の最適化結果から、利用者ごとに異なる粒度の情報を参照できる状態を前提とする。

### 10.4 可視化成果物との接続

本 PoC では、表形式の出力だけでなく、各候補案に対応するルート可視化成果物も出力対象とする。  
可視化成果物は、配送計画担当者および現場責任者が、地理的なルート妥当性や担当範囲の偏りを直感的に確認するために用いる。

そのため、各候補案は、候補案単位の集計表、ルート明細、可視化成果物が相互に対応づく形で管理する。

### 10.5 暫定結果および実行可能解なしの出力方針

暫定結果を出力する場合は、通常の候補案と同じ形式で出力情報を保持しつつ、当該結果が全件配送前提を満たしていないことを明示する。  
具体的には、失敗地点または失敗地点対を除いた結果であることを、実行時条件情報および補助情報で識別可能とする。

一方で、実行可能解なしの場合は、通常の候補案比較結果とは別の状態として扱う。  
この場合は、候補案を返すのではなく、制約条件を満たす解が存在しなかったこと、および見直し対象となる可能性がある条件を確認対象として返す。

### 10.6 出力データ一覧

本 PoC では、数理モデルの解を以下の出力データとして保持する。

- 候補案単位の出力: `candidate_plan_summary`
- ルート明細単位の出力: `candidate_route_detail`
- 可視化単位の出力: `visualization_artifact`

これにより、数理モデルの解を、比較評価、現場確認、可視化確認に利用できる形で出力する。

## 11. 本 PoC における数理モデルの限界

本章では、本 PoC における数理モデルの限界を整理する。  
本 PoC は、実道路ネットワークベースの VRP による配送計画最適化の有効性を検証することを目的としているため、現場運用を完全に再現するものではない。  
そのため、以下のような近似、未考慮事項、解釈上の制約を持つ。

### 11.1 時間に関する限界

本 PoC で用いる所要時間は、OpenStreetMap の道路属性にもとづく静的推定時間であり、実交通を反映した動的時間ではない。  
そのため、渋滞、事故、時間帯混雑、信号待ちなどの影響は直接反映しない。

また、時間計算には補正係数を用いるが、これは外部経路案内サービス等を参照した近似的な調整値であり、実運行時間を保証するものではない。  
したがって、本 PoC の時間制約は、厳密な到着保証ではなく、業務上不自然な解を抑制するための近似的な判定として解釈する必要がある。

### 11.2 距離・道路ネットワークに関する限界

本 PoC では、道路ネットワークの正本として OpenStreetMap を利用する。  
そのため、道路接続、一方通行、通行可否を踏まえた実道路ベースの距離計算が可能である一方、他の地図サービスが持つ道路情報や経路探索結果とは差異が生じる可能性がある。

また、本 PoC では、建物入口、荷捌き位置、私道進入可否、施設内動線などの詳細な現地条件は扱わない。  
したがって、道路ネットワーク上は到達可能であっても、現場感覚では最適でないルートとなる可能性がある。

### 11.3 車両・コストモデルに関する限界

本 PoC では、車両を同一仕様として扱い、容量、固定費、距離比例費を共通パラメータとして定義する。  
そのため、車種差、積載差、車両ごとの固定費差、配送員ごとの熟練差などは扱わない。

また、コストは `車両固定費 × 使用台数 + 距離単価 × 総走行距離` に簡略化しており、燃料費、人件費、車両維持費、高速料金、再配達コスト、遅配リスクなどを厳密に積み上げたものではない。  
したがって、本 PoC のコストは厳密会計値ではなく、候補案比較のための概算指標として解釈する必要がある。

### 11.4 業務運用に関する限界

本 PoC では、単一デポ、単一都道府県内配送、`1台 = 1人`、当日中配送、closed VRP を前提としている。  
そのため、複数デポ運用、都道府県を跨ぐ広域配送、再配達、当日中の配送先変更、複数人乗務、徒歩複合移動などの実運用パターンは対象外である。

また、荷下ろし時間、駐車待ち時間、顧客対応時間などの詳細な現場作業時間は現時点では扱わない。  
そのため、現場作業負荷まで含めた厳密な運行計画ではなく、移動を中心とした配送計画近似として扱う必要がある。

なお、業務開始時刻および業務終了時刻の扱いは、現時点では単一デポに紐づく共通条件として固定的に扱うが、将来的には車両ごと、便ごと、または運用条件ごとに柔軟化できるよう拡張余地を残す。

### 11.5 解釈上の注意

本 PoC の出力は、現場での最終判断を完全自動化するものではなく、配送計画担当者、現場責任者、導入判断者が比較・検討するための候補案である。  
また、前処理不備により一部地点を除外している場合は、数理モデル上は実行可能解であっても、全件配送前提を満たしていないため暫定結果として扱う。

したがって、本 PoC の出力は、実務上の唯一の正解として扱うのではなく、業務判断、データ修正、再実行を前提とした意思決定支援情報として利用する必要がある。
